# 寄存器默认值：芯片上电后，谁替你写了那些你不写的值

## 核心问题

芯片上电后，在你写第一行代码之前，那些寄存器的值是谁写的？

**不是 POR。** POR 只保证所有触发器被强制拉到复位状态。但"复位状态"不等于"0"。那些上电后默认就是 0x00000001 的寄存器，是电路自己"长"出来的——数字设计工程师在 RTL 里逐一指定的上电初始状态。

## 默认值的物理来源

一个触发器在上电后的输出是 0 还是 1，由两个物理机制决定：

**1. 带异步复位端的触发器**
- POR 信号拉低时，所有带异步复位端的触发器被强制拉到 RTL 里指定的复位值
- 绝大多数复位值写成 0，但电源管理/安全相关触发器可能指定为 1
- 由 RTL 里 `FDCE`（复位清0）或 `FDPE`（复位置1）的端口连接定死

**2. 无复位端的触发器**
- 为了省面积、省功耗，综合时未接复位端
- POR 释放后输出随机，取决于热噪声、电源上电斜率、工艺偏差
- 功能安全分析中标记为"未定义"

**手册"默认值"列的物理含义：**
- 0x00000000 → 所有触发器接了异步复位，复位值为 0
- 0x00000001 → bit0 的触发器复位值为 1，其余为 0
- u（undefined）→ 无复位端，POR 后输出不确定

## 为什么默认值不是全 0？

**决策一：上电安全状态必须是"不动作"**
- 绝大多数外设默认"不动作"：PWM 不输出、GPIO 高阻输入、DMA 不传输
- 某些电源管理模块，"关闭"电平在物理上是高电平 → RTL 对应使能位复位值必须指定为 1
- 硅片上的物理需求决定逻辑极性

**决策二：让"最常用的值"成为默认值**
- 数字设计工程师替固件工程师写初始化代码
- PLL 默认倍频系数为 8（最常用）
- Flash 等待周期默认 1（低速 RC 振荡器下够用）
- 固件少写代码，也少一次写错的机会

**决策三：省面积、省功耗，放弃部分复位端**
- 高速数据路径、大型 FIFO、不关心上电初始态的配置寄存器省略复位端
- 折中方案：POR 释放后极短时间内，硬件状态机自动写一次默认值（"硬件自动加载"）
- 代价：需要等若干微秒才能访问，自动加载完成前读随机值

## 固件工程师三种典型陷阱

**陷阱一：把"默认值"当成"没人管的值"**
- 芯片另一版本可能默认值变了（换 IP 供应商、场景假设不同）
- **正确做法**：初始化阶段全部显式写入，不依赖任何默认值

**陷阱二：对"只写"寄存器读默认值**
- 只写寄存器读返回 0 是总线接口逻辑的固定电平，不是真实状态
- 用假的 0 覆盖内部真实状态
- **正确做法**：只写寄存器绝不读-修改-写，用影子变量维护虚拟状态

**陷阱三：硬件自动加载没完成就动手**
- POR 后某些外设需等硬件自动加载完成
- 开发板通常刚好覆盖延迟，换电源芯片后上电斜率变化，随机启动失败
- **正确做法**：遵守手册标明的延迟，bringup 阶段验证不同电源条件

## 固件铁律

1. **不依赖默认值**：初始化阶段所有控制位、配置位全部显式写入
2. **只写寄存器不读**：用影子变量维护虚拟状态，最终完整赋值
3. **上电后验证生存基线**：读芯片ID、复位状态、时钟状态，确认总线可通
4. **注意无复位端寄存器**：手册标"default: u"的位必须显式写入，不能假设为 0

## 总结

> 默认值不是"无人值守"的结果，而是数字设计工程师在 RTL 里一个触发器一个触发器地决定的。它考虑硅片物理安全、固件使用习惯、面积功耗取舍，最终凝固成手册里那一列"Default"数字。
