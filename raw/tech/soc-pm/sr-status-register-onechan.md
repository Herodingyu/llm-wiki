## 核心观点

SR 不是一个简单的"存储单元"，它是**数字系统中硬件与软件之间唯一的、标准化的、实时的状态同步接口**。它解决的根本矛盾是：**硬件是并行的、连续的、事件驱动的，而软件是串行的、离散的、指令驱动的**。

## 为什么必须有 SR？

如果没有 SR，软件怎么知道硬件刚才做了什么？

以 `ADD R0, R1, R2` 为例，CPU 执行需要 3 个时钟周期：
1. 周期 1：读 R1 和 R2
2. 周期 2：ALU 计算
3. 周期 3：结果写入 R0

软件看到的"指令执行完毕"，只是 CPU 保证了结果已写入，但不会告诉你：
- `0x80000000 + 0x80000000` 是溢出还是得到 0
- 减法结果是正、负还是零
- 除法运算中除数是不是零
- 内存访问有没有发生页错误

有了 SR，硬件在 ALU 计算的同时，用一个额外的与门电路在 1 个周期内生成溢出标志位 OV，软件只需读 SR 的第 11 位即可。

**SR 把硬件并行产生的、关于操作结果的所有属性信息，在操作完成的同一个周期内，集中锁存到一个软件可访问的寄存器中**。

## SR 的硬件本质："事件锁存器阵列"

- **通用寄存器**：软件写、硬件读
- **SR**：**硬件写、软件读**的事件锁存器阵列

每个标志位背后对应一个独立的硬件事件源，事件发生时硬件自动置位；软件只能通过特定指令读取或清除，不能像写普通寄存器一样随意修改。

## 经典标志位深度解析

### 算术标志位（所有条件分支指令的基础）

| 标志位 | 名称 | 硬件生成逻辑 | 软件意义 |
|--------|------|-------------|----------|
| Z | 零标志 | Result == 0 | 运算结果为零 |
| N | 负标志 | Result[31] == 1 | 运算结果为负数 |
| C | 进位标志 | 运算产生进位/借位 | **无符号数**运算溢出 |
| O | 溢出标志 | 符号位产生不正确进位 | **有符号数**运算溢出 |

**极其重要的区别**：C 和 O 分别对应无符号数和有符号数的溢出，完全独立。

### 系统标志位

#### 中断使能标志位（I/F）
为什么必须放在 SR 里？为了**原子性的上下文切换**。

响应中断时：
1. 当前 PC 压栈
2. 当前 SR 压栈
3. 自动清除 I 位，禁止嵌套中断

中断返回（IRET）时：
1. 弹出之前的 SR（恢复所有标志位，包括 I）
2. 弹出之前的 PC，继续执行

如果 I 不在 SR 里，上下文切换就不是原子的，会导致中断丢失或嵌套错误。

#### 模式位（M[4:0]）
处理器特权级机制的核心。ARM CPSR 的 5 位模式位定义 7 种处理器模式。模式位只能在特权模式下修改，用户模式下只读 —— 这是操作系统安全模型的基础。

## SR 的读写语义陷阱

### 读 SR：获取当前硬件状态的快照
执行 `MRS R0, CPSR` 时，得到的是**执行这条指令的那个时钟周期，CPU 所有状态的原子快照**。纯副作用操作，不会改变硬件状态。

### 写 SR：不是"覆盖"，是"请求硬件改变状态"
执行 `MSR CPSR, R0` 时，**不是向存储单元写入值，而是向硬件发送状态改变请求**。硬件逐位检查，根据语义执行操作：
- 修改模式位 → 切换处理器模式 + 切换栈指针
- 修改中断使能位 → 立即打开/关闭中断
- 修改算术标志位 → 更新对应锁存器
- 保留位/只读位 → 忽略

**致命陷阱**：
```c
// 错误！读取到修改后写回之间，硬件可能修改了其他标志位
uint32_t cpsr = read_cpsr();
cpsr &= ~(1 << 7);  // 清除 I 位
write_cpsr(cpsr);    // 可能覆盖掉中断返回时修改的 C 标志

// 正确：只修改需要的位域，使用专用位修改指令
enable_irq();  // 编译器内置函数
```

## 现代处理器中 SR 的演变

### RISC-V：彻底抛弃单一 SR
将状态信息分散到不同 CSR：
- `mstatus`：机器模式状态
- `mcause`：异常原因
- `mip`：中断挂起
- `mie`：中断使能

### x86：EFLAGS → RFLAGS 扩展
保留传统算术标志位，增加系统级标志位：IF、VIP、VIF、AC 等。

### 共性趋势：**状态信息与控制信息的分离**
早期处理器把控制位和状态位放在同一个 SR，导致读写语义混乱。现代处理器将纯状态信息和纯控制信息分开到不同寄存器。

## 固件开发中 SR 的常见坑

### 坑1：中断服务程序中修改算术标志位
中断会破坏被中断程序的 SR 算术标志位，如果中断返回时没有正确恢复，会导致条件判断出错。

**解决方案**：中断服务程序入口处保存 SR，出口处恢复 SR。

### 坑2：使用普通 LDR/STR 指令修改 SR
很多处理器不允许用 `LDR/STR` 直接修改 SR，必须使用专用 `MRS/MSR` 指令。错误使用会导致硬件忽略写入或产生未定义指令异常。

### 坑3：依赖未定义的标志位行为
很多指令对标志位的影响是未定义的。如 ARM 的 `MOV` 不影响标志位，`MOVS` 才会影响。不要做任何假设。

## 最佳实践

1. **最小化修改 SR**：只修改必须修改的位，不要一次性写入整个 SR
2. **使用编译器内置函数**：`__enable_irq()`、`__disable_irq()` 等
3. **临界区尽可能短**：只包含必要的寄存器访问
4. **永远不要相信 SR 值会保持不变**：每次使用前重新读取

## 总结

SR 是硬件与软件的"握手协议"：
- 硬件承诺：完成任何操作后，将所有相关状态信息准确锁存到 SR
- 软件承诺：需要知道硬件状态时，只从 SR 读取，不通过其他方式猜测

没有 SR，软件就无法做出任何逻辑判断，无法处理任何异常，无法实现任何复杂功能。
