如果说 CPU 是 SoC 的“大脑”，存储系统是“记忆”，外设子系统是“感官与手脚”，那么互联子系统就是 SoC 内部的“道路、桥梁和交通规则”。很多初学者看 SoC 架构图时，会很自然地关注 CPU、GPU、NPU、DDR、外设接口这些功能模块，却容易忽略连接它们的互联网络。 

但从系统架构角度看，互联子系统并不是简单的“连线”。它决定了数据如何流动，决定了谁能访问谁，决定了系统性能上限，也决定了芯片面积、功耗、时序收敛和可扩展性。一个 SoC 不是把很多 IP 摆在一起就能工作。真正让这些 IP 组成系统的，是互联。

## 一、为什么 SoC 需要互联？ 

先回到最基本的问题：  SoC 内部到底在发生什么？ 

本质上只有一件事：  多个计算、存储和外设模块，在共享数据和状态。 

 CPU 要从 Flash 或 DDR 取指令； 

 CPU 要读写 SRAM； 

 DMA 要把外设数据搬到内存； 

 GPU/NPU 要从 DDR 读取大块数据； 

 外设要通过寄存器被 CPU 配置； 

  多核 CPU 之间可能还要共享缓存一致的数据。  这些行为都可以抽象成两个动作： 

读：从某个地方拿数据。 

写：把数据放到某个地方。 

当 SoC 中存在多个数据请求方和多个数据响应方时，系统必须提供一套机制，让它们安全、有序、高效地完成通信。 

这里的  数据请求方  通常叫 Master 或 Initiator，例如 CPU、DMA、GPU、NPU、显示控制器、以太网控制器。  数据响应方  通常叫 Slave 或 Target，例如 SRAM、DDR 控制器、寄存器模块、外设控制器、Flash 控制器。 

互联子系统要解决的问题，就是： 

 谁可以访问谁？ 

 访问地址如何路由？ 

 多个请求同时发生时谁先走？ 

 数据返回时如何找到原来的请求方？ 

 不同位宽、频率、协议之间如何转换？ 

 系统拥塞时如何保证关键业务不被饿死？ 

 多核共享数据时如何保持一致性？ 

所以，互联不是线，而是 SoC 内部的交通系统。 


## 二、互联子系统的核心矛盾：连接越多，复杂度越高 

如果只有一个 CPU 和一个 SRAM，事情很简单，一根简单总线就够了。  CPU 发地址，SRAM 返回数据  。但现实 SoC 不是这样。一个稍复杂的 SoC 可能同时有： 

多个 CPU 核； 一个或多个 DMA； 图像处理单元； AI 加速器； 显示控制器； 视频编解码器； DDR 控制器； 片上 SRAM； Flash 控制器； 大量低速外设。 

这时系统会出现一个核心矛盾：  每个模块都想高效访问资源，但芯片不可能给每一对模块都铺一条专用高速通道。 

如果全部点对点直连，性能很好，但连线爆炸、面积爆炸、验证复杂度爆炸。如果全部共享一根总线，结构简单，但所有请求都会抢同一条路，性能很快遇到瓶颈。 

因此，互联子系统的设计，本质上是在几个目标之间做平衡： 

性能； 面积；功耗； 时序； 可扩展性； 实时性；软件可见性  ； 验证复杂度  。  不同互联方式，本质上对应不同规模和性能目标的 SoC。

## 三、第一种互联：总线，最简单的共享道路 

总线是最容易理解的互联方式。可以把它想象成一条公共道路。多个主设备和多个从设备都接在这条路上。某一时刻通常只有一个主设备获得使用权，然后发起读写访问。 

总线的优点非常明显： 

 结构简单； 

 面积小； 

 协议容易理解； 

 适合低成本、小规模系统； 

  软件和调试模型直观。 

很多 MCU 或小型 SoC 中，AHB、APB 这类总线结构仍然非常常见。 

但总线的问题也很明显： 

 所有访问共享同一条通路； 

 主设备变多后仲裁压力增加； 

 高带宽模块容易堵塞低带宽模块； 

 系统规模扩大后频率和时序都变难； 

  同时访问多个从设备的并行性较弱。  举个例子，如果 CPU、DMA、显示控制器都要访问同一条共享总线，而显示控制器又必须稳定从内存取图像数据，那么 DMA 的大块搬运就可能影响显示刷新，导致画面撕裂或卡顿。 

因此，总线适合简单系统，但不适合所有模块都高并发、高带宽的复杂 SoC。 


## 四、第二种互联：交叉矩阵，让多个访问并行发生 

交叉矩阵可以理解成一个“交通枢纽”。它不是一条共享道路，  而是尝试在多个主设备和多个从设备之间建立多条可并行的路径  。如果 CPU 正在访问 SRAM，而 DMA 正在访问 DDR，只要目标不同、资源不冲突，交叉矩阵就可以让它们同时进行。 

它相比总线的优势是： 

 并行性更好； 

 带宽利用率更高； 

 多个 Master 可以同时访问不同 Slave； 

 适合中等规模 SoC 的主系统互联。  但代价也随之上升： 

 硬件面积更大； 

 仲裁逻辑更复杂； 

 连接数量随 Master 和 Slave 增加而快速增长； 

 时序收敛更难； 

  验证状态空间变大。  这就像城市交通从“一条主路”升级成“立交桥”。通行能力更强，但设计、施工和维护成本也更高。在实践中，很多中型 SoC 会采用局部交叉矩阵结构，例如 CPU、DMA、SRAM、DDR 控制器之间使用性能较高的互联，而低速外设仍然挂在独立的低速总线上。 

这体现了一个重要原则： 

不是所有模块都应该接入最高性能互联。 

高性能互联应该服务于真正需要带宽和低延迟的模块。 


## 五、第三种互联：NoC，让 SoC 进入片上网络时代 

当 SoC 继续变大，模块越来越多，传统总线和交叉矩阵都会遇到扩展性问题。这时就出现了 NoC，也就是 Network on Chip，片上网络。 

NoC 的思想很直观：  既然芯片里有很多模块，那就不要再把通信看成一根线，而要看成一个网络。  每个 IP 通过网络接口接入片上网络，数据被封装成事务或数据包，在路由节点之间转发，最终到达目标模块。 

NoC 的优势在于： 

可扩展性强； 

 适合多核、多加速器、大规模 SoC； 

 可以通过路由和拓扑优化带宽； 

 更容易做分区、QoS 和流量管理； 

  长距离连线可以被分段，有利于时序收敛。  比如手机 SoC、AI SoC、车载 SoC 中，CPU、GPU、NPU、ISP、视频编解码、DDR 控制器之间的数据流非常复杂。如果仍然用一条共享总线，很容易成为瓶颈。NoC 可以把这些访问变成多个网络流，并通过路由、仲裁和优先级机制进行管理。 

但 NoC 也不是免费的午餐。它会带来： 

 更高的设计复杂度； 

 更复杂的延迟分析； 

 更复杂的死锁、拥塞和 QoS 验证； 

 更多缓冲和路由逻辑； 

  更高的软件性能调优门槛。  因此，NoC 不是为了“高级”而使用，而是在系统规模和并发访问复杂度达到一定程度后，传统互联方式已经不够用时的自然选择。 


## 六、第四种互联：缓存一致性总线，解决“多核共享数据”的问题 

前面讲的总线、交叉矩阵和 NoC，主要解决的是“如何访问”。但在多核系统里，还有一个更麻烦的问题：  如果多个 CPU 都缓存了同一份内存数据，那么谁看到的数据才是最新的？ 

比如 CPU0 修改了某个变量，但这个变量还停留在 CPU0 的 L1 Cache 中，尚未写回内存。此时 CPU1 如果从自己的 Cache 或内存里读取旧值，就会产生错误。这就是缓存一致性问题。 

缓存一致性互联要解决的是： 

一个核写了共享数据，其他核如何知道？ 

 一个核读共享数据时，如何拿到最新值？ 

 多个核同时访问同一缓存行时，状态如何转换？ 

 DMA 或外设访问内存时，是否需要和 CPU Cache 保持一致？ 

实践中，很多多核 SoC 会引入 ACE、CHI 等一致性互联协议，或者使用系统级缓存、Snoop Filter、一致性管理单元来降低一致性维护成本。 

缓存一致性互联不是为了提高单次访问速度，而是为了让多核共享内存编程模型成立。  它解决的是“正确性”问题，然后才是性能问题。 


## 七、互联子系统不能只有一种路：必须分层设计 

真正的 SoC 互联通常不是一种结构打天下，而是分层的。可以把 SoC 互联想象成城市道路系统： 

 高速路连接核心城区和机场； 

 主干道连接不同城区； 

 支路连接小区和街道； 

 小区内部道路只服务低速通行。SoC 里也是一样。 


### 1\. 高性能主互联 

用于 CPU、GPU、NPU、DDR 控制器、片上 SRAM、高速 DMA 等模块。 

特点是高带宽、低延迟、可能支持乱序、突发传输、QoS、一致性等。这里最关心的是性能和并发。 


### 2\. 中速共享互联 

用于一些数据吞吐量中等的模块，例如 USB、SDIO、摄像头接口、显示控制器、音频接口等。它们不一定需要和 CPU 一样的低延迟，但可能需要稳定带宽。这里最关心的是带宽保障和系统隔离。 


### 3\. 低速外设互联 

用于 UART、I2C、SPI、GPIO、Timer、Watchdog、PWM 等外设寄存器访问。 

这类模块通常不需要高带宽，但需要简单、可靠、低功耗。所以常见做法是通过桥接器从高性能主互联转到 APB 等低速外设总线。 

这和前面外设子系统的分析正好衔接：  子系统之所以常挂在低速总线上，是因为它们的访问模式主要是寄存器配置，而不是持续高带宽数据流。 

因此，互联分层的本质是：  让不同类型的流量走不同等级的道路。 


## 八、数据通路和控制通路要分开看 

做 SoC 架构时，非常重要的一点是区分两类访问： 

数据通路访问。 

控制通路访问。 

数据通路关注大吞吐、大块搬运、持续带宽。例如： 

 NPU 读取模型权重； 

 显示控制器从 DDR 取 framebuffer； 

 摄像头数据写入内存； 

 以太网 DMA 收发数据包； 

  CPU 访问 DDR 和 Cache。  控制通路关注寄存器配置、状态读取、中断清除。例如： 

 CPU 配置 UART 波特率； 

 CPU 设置 DMA 源地址和长度； 

 CPU 读取外设状态寄存器； 

 CPU 清除中断标志位； 

 CPU 配置时钟和复位寄存器。 

这两类访问的需求完全不同。数据通路需要带宽和并发。控制通路需要简单和确定性。  如果把所有访问都放在同一条高性能复杂互联上，面积和功耗会浪费。 

  如果把高吞吐数据也放在低速控制总线上，性能会崩溃。  所以实践中常见结构是： 

高带宽数据通路走 AXI、NoC、主系统互联；低速控制通路走 APB 或寄存器总线； 

  中间通过桥接器做协议转换和访问隔离。  这也是 SoC 互联设计里非常基础但非常重要的实践原则。 


## 九、互联运行频率：不是所有模块都跑同一个时钟 

芯片中不同区域、不同模块的速度和时序需求并不相同。从实践看，SoC 中常常存在多个时钟域： 

 CPU 高频； 

 DDR 控制器和 PHY 有自己的时钟； 

          NoC 或主互联一个频率； 

          外设总线较低频； 

          低功耗域可能运行在更低频率； 

          RTC 甚至可能使用 32 kHz 低速时钟。 

这带来一个问题：  不同时钟域之间如何可靠传输数据？ 

这就是 CDC，也就是 Clock Domain Crossing，时钟域跨越。互联设计中经常需要： 

异步桥； 

          同步 FIFO； 

          握手机制； 

          时钟门控； 

           复位同步； 

          跨域中断同步。 

如果这些问题处理不好，就可能出现非常隐蔽的 bug：仿真中不容易复现，芯片上偶发死锁、数据错误、状态机卡死。所以互联子系统不仅要解决“连得上”，还要解决“跨频率、跨电源域后仍然可靠”。 

不同节奏的系统之间，必须通过明确的同步协议交换信息。 


## 十、系统总线：数据总线与寄存器总线 


### 1\. 数据总线：搬大数据 

数据总线服务于高吞吐的数据流。它关心： 

带宽； 

          突发传输； 

                  读写并发； 

                  Outstanding 数量； 

                  乱序返回； 

                  QoS； 

                  延迟； 

                  仲裁策略。 

例如 NPU 读取 DDR 中的权重  和激活值，显示控制器持续读取图像缓冲区，DMA 搬运外设数据，这些都属于数据通路。 

数据总线的设计目标是：尽量让大块数据流顺畅通过。 


### 2\. 寄存器总线：管控制状态 

寄存器总线服务于配置和状态访问。它关心： 

简单； 

                  可靠； 

                  低功耗； 

                  易调试； 

                  地址映射清晰； 

                  错误响应明确。 

比如 CPU 写一个寄存器启动 DMA，读一个寄存器查看 UART 是否收到数据，写一个寄存器清除中断。这些访问量不大，但对系统控制至关重要。 

寄存器总线的设计目标不是高吞吐，而是让软件能够清楚地控制硬件。 


## 十一、一个实践例子：摄像头数据进入内存 

我们用一个实际场景来串起来理解互联子系统。

假设一个 SoC 要支持摄像头采集图像，并把图像送到内存，再交给 NPU 做识别。 

整个过程可能是： 

摄像头接口接收外部图像数据； 

  摄像头控制器通过 DMA 写入 DDR； 

  NPU 从 DDR 读取图像数据；  

  NPU 把推理结果写回内存； 

  CPU 读取结果并做决策； 

  显示控制器可能再从 DDR 读取画面显示出来。 

这个场景里，互联子系统至少要处理几类流量： 

摄像头 DMA 写 DDR； 

                          NPU 读写 DDR； 

                          CPU 访问 DDR； 

                          显示控制器读 DDR； 

                          CPU 配置摄像头、DMA、NPU、显示模块的寄存器； 

                          中断控制器通知 CPU 任务完成。 

如果所有这些访问都挤在一条简单总线上，系统很容易堵塞。更合理的设计是： 

摄像头 DMA、NPU、显示控制器等高带宽模块接入主系统互联或 NoC； 

                  DDR 控制器作为关键共享资源，需要仲裁和 QoS； 

                  CPU 的控制访问通过寄存器总线进入各模块； 

                  低速配置走 APB，高速数据走 AXI/NoC； 

                  关键实时流量，例如显示读 DDR，可以设置更高优先级或带宽保障； 

                  中断路径从各模块汇聚到中断控制器，再通知 CPU。 

这就是互联子系统的真实价值：  它不是单纯连接模块，而是在多个并发数据流之间建立秩序。 


## 十二、互联设计最容易踩的坑 


### 1\. 只看峰值带宽，不看并发场景 

很多设计会计算单个模块需要多少带宽，却忽略多个模块同时工作时的叠加压力。 

例如摄像头写 DDR、显示读 DDR、NPU 读写 DDR、CPU 跑程序可能同时发生。 

互联设计必须做  系统级带宽建模，而不是只看单点峰值  。 


### 2\. 控制通路和数据通路混在一起 

如果高带宽数据流阻塞了寄存器访问，CPU 可能无法及时配置或停止某个模块，甚至影响异常处理。因此控制路径通常需要保持足够可达性和确定性。 


### 3\. 忽略 QoS 

有些模块对延迟不敏感，比如后台 DMA 搬运。有些模块对延迟非常敏感，比如显示、音频、实时控制。  互联如果没有优先级和带宽保障机制，就可能出现平均性能看起来不错，但用户体验很差的问题。 


### 4\. 跨时钟域处理不严谨 

异步桥、同步 FIFO、复位同步、握手协议都不是可有可无的细节。 

很多芯片级偶发问题，最后都可能追到   CDC 或 reset domain crossing  。 


### 5\. 地址映射混乱 

互联还负责地址解码和路由。如果地址空间规划不清晰，软件、验证、调试都会痛苦。好的地址映射应该层次清楚、留有扩展空间、权限边界明确，并能支持安全域和非安全域隔离。 


## 十三、从架构师视角如何设计互联子系统？ 

会先问几个更底层的问题： 

 系统有哪些 Master？ 

 系统有哪些 Slave？ 

 哪些访问是高带宽数据流？ 

 哪些访问是低速控制流？ 

 哪些模块会同时访问 DDR？ 

 哪些业务有实时性要求？ 

 哪些模块需要缓存一致性？ 

 哪些模块跨时钟域或电源域？ 

 哪些路径需要安全隔离？ 

 哪些外设只需要寄存器访问？ 

 未来是否要扩展更多 IP？ 

然后再决定： 

 主互联用总线、交叉矩阵还是 NoC； 

 低速外设是否通过 APB Bridge 下挂； 

 数据总线和寄存器总线如何分离； 

 DDR 访问如何做仲裁和 QoS； 

 多核是否需要一致性互联； 

 跨时钟域是否需要异步桥或 FIFO； 

  地址映射和权限控制如何规划。  互联设计的关键不是选择一个“看起来高级”的协议，而是让系统流量和互联结构匹配。 

SoC 的难点不只是把 CPU、存储、外设、加速器集成到一颗芯片里。真正的难点是让它们在同一颗芯片里高效协同。互联子系统正是这种协同的基础。它让 CPU 能访问存储，让 DMA 能搬运数据，让外设能被配置，让加速器能共享内存，让多核之间能保持一致，让不同速度、不同协议、不同电源域的模块能组成一个可工作的系统。所以，互联不是 SoC 架构里的背景板。它是 SoC 从“IP 拼图”变成“完整系统”的关键。一个优秀的互联子系统，应该让高带宽数据流跑得快，让低速控制访问走得稳，让关键实时业务不被饿死，让多核共享数据保持正确，让未来扩展不至于推倒重来。从这个角度看，互联子系统就是 SoC 内部真正的“操作系统级基础设施”。它不直接完成应用功能，却决定所有功能能否顺畅运行。
