很多人刚接触嵌入式 Linux 时，都会觉得系统启动像一件很"自然"的事：

上电，串口刷日志，最后进 Shell。

看起来系统像是"一口气自己起来了"。

但如果你真正做过板级调试、移植或启动排障，很快就会发现：

Linux 启动从来不是一步完成，而是一条很长的接力链。

这条链上任何一环出问题，系统都可能起不来：

• 可能连 U-Boot 都看不到
• 可能卡在 DDR 初始化
• 可能内核已经启动，却挂不上根文件系统
• 也可能 rootfs 没问题，但用户空间没真正起来

所以，想搞懂 Linux 启动，第一步不是背命令，而是先建立一个整体视角。

## 一. 一句话先讲明白

如果只记一句话，可以先记这个：

Linux 启动本质上是一场"逐级接力"：BootROM 拉起第一棒，Bootloader 拉起内核，内核挂起 rootfs，最后用户空间把系统真正跑起来。

你也可以把它简单理解成 4 个阶段：

1. 芯片先找到从哪里启动
2. Bootloader 把内核和设备树准备好
3. 内核完成自身初始化并挂载根文件系统
4. 第一个用户空间进程启动，系统才真正进入可用状态

这就是 Linux 启动最核心的主线。

## 二. 第一阶段：上电之后，谁先开始工作？

很多人会以为系统上电后先跑的是 U-Boot。

其实通常不是。

上电后，第一个真正执行的，往往是 SoC 内部固化的启动代码，也就是：

**BootROM**

它的职责很简单：

• 判断当前从哪个介质启动
• 去该介质上找第一阶段镜像
• 把镜像搬到指定位置
• 然后跳转过去执行

所以 BootROM 并不负责把 Linux 启动完成，它只负责：

**把第一棒交出去。**

如果这一步就失败了，后面什么都不会发生。

这也是为什么有些板子启动失败时，连 U-Boot 提示都看不到。

## 三. 第二阶段：为什么还需要 SPL / TF-A / U-Boot？

因为 BootROM 很小，能力有限。

它只能做最基础的事，真正复杂的初始化还得交给后面的引导程序。

这一阶段常见的角色包括：

• SPL
• TF-A
• U-Boot

它们大致负责这些工作：

• 初始化 DDR
• 配置时钟
• 准备运行环境
• 加载更大的 bootloader
• 最终加载 Linux 内核和设备树

为什么 DDR 初始化总在很早阶段出现？

因为如果 DDR 没起来，后面大镜像就没地方放。

所以很多"Linux 起不来"的问题，根本不是 Linux 本身的问题，而是：

**系统连跑 Linux 的基本条件都还没准备好。**

## 四. 第三阶段：U-Boot 到底做了什么？

到了 U-Boot，整个启动流程才真正进入大家熟悉的范围。

U-Boot 最核心的任务其实就三件事：

• 决定从哪里启动
• 把什么加载到内存
• 用什么参数把内核拉起来

比如：

• 从 SD/eMMC/NAND/SPI NOR/TFTP 加载镜像
• 加载 kernel
• 加载 dtb
• 准备 bootargs
• 执行 bootm / booti / bootz

所以你可以把 U-Boot 理解成：

**Linux 启动前的总调度员。**

它不负责运行 Linux 的业务逻辑，但它决定：

• 启动介质对不对
• 镜像对不对
• 设备树对不对
• 启动参数对不对

这也是为什么很多启动失败，最终根因其实都落在 U-Boot 这一层。

## 五. 第四阶段：内核起来后，为什么系统还不算"进系统"？

很多人觉得执行了 bootm 或 booti 之后，Linux 就算起来了。

其实这只是开始。

内核接管 CPU 后，还要继续完成很多初始化：

• 建立内存管理
• 初始化中断和调度器
• 解析设备树
• 初始化驱动
• 识别串口、块设备、网络设备等硬件

这时候你通常已经能看到 Linux 日志了，

但系统还远没有真正进入可用状态。

因为内核只是先把自己站稳。

## 六. 第五阶段：为什么 rootfs 这么关键？

内核起来之后，下一件大事就是：

**挂载根文件系统 rootfs。**

这一步非常关键，因为 Linux 不是只靠一个内核就能正常工作的。

真正的用户空间程序、库、配置文件、脚本，都在 rootfs 里。

所以一旦挂不上 rootfs，就会看到非常经典的错误：

```
VFS: Cannot open root device
Waiting for root device
Kernel panic - not syncing: VFS: Unable to mount root fs
```

这类问题的本质不是"内核没起来"，而是：

**内核起来了，但找不到自己的家。**

## 七. 第六阶段：为什么还要关心 init？

很多人以为 rootfs 挂上就结束了。

其实还差最后一棒：

**启动第一个用户空间进程。**

它通常是：

```
/sbin/init
systemd
busybox init
```

这个进程是整个用户空间的起点。

如果它起不来，系统同样会报错，甚至直接 panic。

所以"进系统"真正意味着什么？

不是只看到内核日志，

而是：

• rootfs 挂上了
• init 跑起来了
• 用户空间服务开始启动了

到这一步，系统才真正可用。

## 八. 最后给一个最实用的判断框架

以后再遇到 Linux 启动问题，先别乱改配置，先判断卡在哪一层：

**看不到任何输出**

先怀疑 BootROM、启动介质、供电、时钟、DDR 早期阶段。

**能看到 U-Boot，但拉不起内核**

先看镜像、设备树、bootcmd、bootargs、启动命令。

**内核日志出来了，但最后报 VFS**

先看 rootfs、root=、分区、存储驱动。

**rootfs 挂上了，但进不了系统**

先看 init、用户空间、文件系统完整性。

你会发现，启动问题一旦按层拆开，复杂度会立刻下降很多。

## 结尾

Linux 启动看起来像"一次启动"，本质上却是一场多阶段接力：

• BootROM 交出第一棒
• Bootloader 把内核拉起来
• 内核把硬件和 rootfs 准备起来
• 用户空间再把整个系统真正跑起来

所以，理解 Linux 启动最重要的，不是死记每条命令，而是先建立这个全景图：

**从上电到进系统，不是一步，而是分层接力。**

只要这个模型建立起来，后面你再去看 BootROM、U-Boot、Kernel、rootfs、init，整个思路就会清楚很多。
