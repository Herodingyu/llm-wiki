原创alltowinealltowine芯片系统成长记

如果说处理器子系统是 SoC 的"大脑"，互联子系统是"道路和桥梁"，外设子系统是"感官和手脚"，那么存储子系统就是 SoC 的"记忆系统"。但这个"记忆系统"并不是简单地放几块 SRAM、接一颗 DDR、挂一个 Flash 就结束了。

真正的存储子系统设计，要回答的是一个更底层的问题：

数据应该放在哪里，怎样被快速、可靠、低功耗、可预测地访问？

在 SoC 中，几乎所有模块都离不开存储：

CPU 要从存储中取指令、读写变量；

DMA 要把外设数据搬进内存；

NPU/GPU 要读取大块模型、图像和特征数据；

系统启动时要从 Flash 取 Boot Code；

安全模块要读取密钥或配置；

异常发生时还要保存现场、日志和状态。

所以，存储子系统不是某个单独模块，而是支撑整个 SoC 运行的基础设施。

## 一、为什么 SoC 需要存储层级？

我们先不谈 SRAM、DRAM、Flash、Cache 这些名词，而从一个最基本的问题开始：

计算为什么需要存储？

因为处理器和加速器并不是凭空计算。它们每一步都需要：

取指令；

读取输入数据；

保存中间结果；

写回最终结果；

保存系统状态。

也就是说，计算本质上是"数据被不断读取、加工、再写回"的过程。但这里马上出现一个矛盾：

计算单元越来越快，而存储访问速度、容量、成本和功耗不可能同时最优。

理想情况下，我们希望存储同时具备：

速度像寄存器一样快；

容量像硬盘一样大；

功耗像 SRAM 一样低；

成本像外部 Flash 一样便宜；

掉电后还能保存数据；

还能被多个模块同时访问。但现实中没有一种存储介质可以同时满足这些要求。

用不同类型、不同速度、不同容量、不同成本的存储组成层级结构，让最常用的数据离计算单元最近，让大容量数据放在更远但更便宜的位置。

这就是存储层级的本质。

二、存储子系统的核心矛盾：速度、容量、成本、功耗不可兼得

SoC 存储设计的核心矛盾，可以概括为四个词：速度、容量、成本、功耗。

越靠近 CPU 的存储，通常速度越快，但容量越小、面积越贵。比如寄存器和 L1 Cache 访问非常快，但不可能做得很大。 片上 SRAM 速度快、功耗相对可控，但面积成本高。DDR 容量大，但访问延迟高，功耗和带宽竞争明显。Flash 可以掉电保存数据，但写入慢、擦除复杂、寿命有限。

所以存储子系统设计不是简单选一种"最好的存储"，而是构建一个组合：

最热的数据放在寄存器和 Cache；

实时关键数据放在片上 SRAM 或 SPM；

大容量运行数据放在 DDR；

启动代码、固件和长期数据放在 Flash；

安全配置、校准信息、芯片身份信息放在 OTP 或 eFuse。

## 三、存储层级：离计算越近，越快也越贵

可以把 SoC 的存储系统想象成一个仓储体系。

CPU 寄存器像手边的工具，最快，但数量极少。

L1/L2 Cache 像办公桌抽屉，常用资料随手可取。

片上 SRAM 像办公室资料柜，容量更大但仍然很近。

DDR 像公司仓库，容量大但取东西要走流程。

Flash 像档案室，能长期保存，但取写速度慢。

外部存储像远程仓库，容量更大，但访问代价更高。这个类比背后有一个实践原则：

数据越常用、越实时、越频繁访问，就越应该靠近计算单元。

举几个例子：

中断服务程序的关键代码，可以放在片上 SRAM 中，避免外部存储访问延迟。

DSP 或 NPU 的内层循环数据，可以放在本地 SRAM 或 ScratchPad Memory 中。

图像帧、模型参数、音视频缓存等大块数据，可以放在 DDR 中。

BootLoader、固件镜像、配置表，可以放在 Flash 中。

芯片 ID、安全密钥哈希、校准参数，可以放在 OTP/eFuse 中。

因此，存储层级不是教材概念，而是 SoC 性能、功耗和实时性的基础。

四、Cache：用硬件自动捕捉"局部性"

Cache 是存储子系统里最经典的设计。

它基于两个非常重要的程序运行规律：

时间局部性：刚访问过的数据，很可能很快再次被访问。

空间局部性：访问某个地址后，很可能访问它附近的地址。

比如 CPU 执行一段循环代码时，会反复访问同一段指令和一组变量。

如果每次都去 DDR 取，效率会很低。Cache 的做法是：把最近用过的一小块数据自动缓存到离 CPU 更近的地方。从软件角度看，Cache 通常是透明的。程序还是访问同一个地址，但硬件会判断数据在不在 Cache 中：如果在，就是 Cache Hit，访问很快；如果不在，就是 Cache Miss，需要从下一级存储取回来。Cache 的价值是：

让大部分访问命中快速小容量存储；降低对 DDR 的访问压力；提高 CPU 平均性能；

隐藏一部分存储访问延迟。

但 Cache 也带来问题：访问延迟不完全确定；多核之间需要一致性维护；DMA 与 Cache 共享内存时可能出现数据不一致；实时系统中 Cache Miss 可能导致抖动。因此，Cache 非常适合通用计算，但在硬实时场景中需要谨慎使用。

五、SPM：让软件显式管理的"确定性存储"

SPM 可以理解为一块由软件显式管理的片上 SRAM。它和 Cache 最大的区别是：Cache 由硬件自动管理，SPM 由软件或编译器显式管理。

Cache 的好处是方便，软件不用管。 SPM 的好处是确定，软件知道数据一定在哪里。比如在音频处理、基带处理、实时控制、DSP 算法中，系统往往更关心"最坏情况下也能按时完成"，而不是平均性能好不好。这时 SPM 就很有价值。软件可以把关键代码、滤波器系数、输入输出缓冲区提前放入 SPM。计算时直接访问 SPM，延迟稳定，不会出现 Cache Miss。处理完后再通过 DMA 把数据搬回 DDR 或外部存储。这体现了一个重要取舍：Cache 追求平均性能，SPM 追求确定性。

对于通用处理器，Cache 更友好； 对于实时 DSP、NPU 本地缓冲、硬实时控制，SPM 更可控。

## 六、SRAM：片上快速工作区

SRAM 是 SoC 中非常常见的片上存储。

它的特点是：访问快；不需要刷新；适合随机访问；可直接作为片上工作区； 但面积大，容量不宜太大。片上 SRAM 常见用途包括：

Boot 阶段临时执行空间；中断向量表；关键代码段；DMA 缓冲区；多核共享小容量数据；低功耗模式下保留状态；NPU/DSP 本地数据缓存。

举个实践例子：很多 SoC 上电后，第一段 Boot ROM 代码会先运行，然后把下一阶段 BootLoader 或关键初始化代码搬到 SRAM 中执行。这样可以避免外部 Flash 或 DDR 尚未初始化时无法执行的问题。

SRAM 的价值不是容量，而是"快、近、可控"。

## 七、DRAM/DDR：大容量运行内存

DRAM，特别是 DDR，是复杂 SoC 中承载大容量运行数据的核心。它适合放：操作系统； 应用程序； 图像帧缓冲； 音视频数据； AI 模型权重； NPU/GPU 中间结果； 网络包缓冲； 大块 DMA 数据。DDR 的优势是容量大、带宽高、单位容量成本较低。

但 DDR 也有明显限制：访问延迟比片上存储高；多个模块同时访问会竞争带宽；功耗较高； 控制器和 PHY 复杂； 初始化流程复杂； 时序、校准和信号完整性要求高。因此，DDR 不是"无限大且无限快的内存池"。

在 SoC 实践中，DDR 往往是最容易成为系统瓶颈的共享资源之一。例如摄像头写入图像、NPU 读取图像、Display 读取 framebuffer、CPU 运行程序，这些都可能同时访问 DDR。这时存储子系统不能只看 DDR 本身，还必须结合互联子系统的仲裁、QoS、带宽预算一起设计。

## 八、Flash：保存代码和长期数据

Flash 的核心价值是掉电不丢数据。它常用于保存：BootLoader； 固件镜像； 配置参数； 文件系统； 日志； 校准数据； 用户数据。Flash 又可以分为不同类型，例如 Nor Flash、Nand Flash、eMMC、UFS 等。不同类型在容量、速度、接口复杂度、可靠性和成本上各有取舍。

从 SoC 启动角度看，Flash 非常关键。因为芯片上电后，CPU 总要从某个地方拿到第一段可执行代码。这个地方可能是片上 ROM，也可能通过 Boot ROM 去读取外部 Flash，再把代码搬到 SRAM 或 DDR 中执行。

Flash 的问题在于：读写速度不如 SRAM/DDR； 擦除粒度较大； 写入寿命有限；

需要坏块管理或磨损均衡；接口控制器可能比较复杂。所以 Flash 更像"长期档案室"，不是高频运行工作区。

九、OTP/eFuse：只写一次的安全与身份存储

OTP 或 eFuse 是一种特殊存储。它通常只能写入一次，或者写入后不能随意修改。这类存储常用于：芯片唯一 ID； 安全启动配置； 密钥哈希； 生命周期状态；

调试口锁定； 出厂校准参数； 加密配置；防回滚版本号。它的价值不在于容量，而在于不可篡改性和安全属性。例如安全启动中，Boot ROM 可以读取 OTP 中的公钥哈希，用它验证外部 Flash 中固件的签名。如果验证通过，系统继续启动；如果验证失败，系统拒绝运行。这说明存储子系统不仅影响性能，也影响安全。

## 十、存储控制器：把存储器变成系统资源

很多人容易把存储理解成"几块 memory macro"。但在 SoC 中，真正让存储可用的是存储控制器。

存储控制器要解决的问题包括：地址译码；读写时序控制； 访问仲裁； 突发传输； 刷新管理； ECC 校验；低功耗管理；错误上报； 初始化和校准； 总线协议适配。例如 DDR 控制器不仅要处理读写请求，还要负责 DDR 初始化、刷新、命令调度、Bank 管理、PHY 训练、带宽优化和功耗状态切换。

Flash 控制器可能要处理页读写、块擦除、ECC、坏块管理、DMA 传输和启动映射。所以，存储控制器的本质是：把复杂的存储介质细节封装起来，对 SoC 内部提供统一、可访问、可管理的存储资源。

十一、存储映射：让软件看到一个统一地址空间

SoC 中的 CPU 通常通过地址访问资源。所以存储子系统必须回答一个问题：

每一段地址对应哪里？比如：

0x0000_0000 可能是 Boot ROM；

0x2000_0000 可能是片上 SRAM；

0x4000_0000 可能是外设寄存器；

0x8000_0000 可能是 DDR；

某段地址可能映射到 Flash 或片上安全存储。这种地址规划就是存储映射。

存储映射的价值在于：

让软件用统一方式访问不同存储；

让互联子系统根据地址完成路由；

让权限控制模块判断访问是否合法；

让调试工具能理解系统资源分布；

让启动流程能够从固定地址开始执行。文档中也提到存储器映射与重映射，这在 SoC 启动和异常处理中非常重要。

十二、实践例子：摄像头图像如何进入存储并被处理？

我们用一个实际场景来理解存储子系统。假设一个 SoC 要从摄像头采集图像，然后交给 NPU 做识别，同时还要把图像显示到屏幕上。

流程大概是：摄像头接口接收图像数据； ISP 做基础图像处理； Camera DMA 把图像帧写入 DDR；NPU 从 DDR 读取图像数据进行推理；Display Controller 从 DDR 读取 framebuffer 刷屏； CPU 通过中断得知采集完成，并读取处理结果。这个流程中，存储子系统承担了几个关键职责。

### 1. DDR 是共享数据中心

摄像头写入，NPU 读取，Display 读取，CPU 也可能访问。因此 DDR 不只是"内存条"，而是多个模块共享的数据中心。

这时需要考虑：带宽是否够； 读写是否冲突； 是否需要双缓冲或环形缓冲； 显示读是否有实时优先级； NPU 大量读写是否会影响摄像头写入；DDR Controller 和互联是否支持 QoS。

### 2. SRAM 可用于关键缓冲或控制数据

有些控制结构、描述符、关键中断代码可以放在片上 SRAM。比如 DMA descriptor 放在 SRAM，可以降低访问延迟。 关键中断服务程序放在 SRAM，可以提高响应确定性。小块临时数据放在 SRAM，可以减少 DDR 访问。

### 3. Cache 会影响 CPU 看到的数据

如果 Camera DMA 把图像写入 DDR，而 CPU 之前缓存过同一段地址，那么 CPU 可能读到旧数据。因此软件需要做 Cache 维护，例如 invalidate 或 clean。 或者系统使用一致性互联，让 DMA 与 Cache 保持一致。 或者把 DMA 缓冲区映射成 non-cacheable 区域。这就是存储子系统和互联子系统耦合的典型问题。

### 4. Flash 保存算法和固件

NPU 推理模型、ISP 参数、固件镜像可能保存在 Flash 中。系统启动后，BootLoader 或操作系统把它们加载到 DDR 或 SRAM 中使用。所以 Flash 是长期存储，DDR 是运行存储，SRAM/Cache 是快速工作存储。

这个例子说明：存储子系统不是一块内存，而是一套围绕数据生命周期设计的系统。数据从哪里来，放到哪里，谁来读，谁来写，什么时候失效，是否需要一致性，是否需要长期保存，这些都属于存储子系统设计。

## 十三、存储子系统最容易踩的坑

### 1. 只看容量，不看带宽

很多设计会说"DDR 容量够大"，但真正的问题可能是带宽不够。摄像头、显示、NPU、CPU 同时访问 DDR 时，容量不是瓶颈，带宽和延迟才是瓶颈。

### 2. 忽略最坏情况延迟

Cache 平均性能很好，但 Cache Miss 会导致访问延迟变长。如果系统有硬实时任务，就必须分析最坏情况，而不是只看平均值。

### 3. DMA 与 Cache 一致性处理不清楚

这是嵌入式系统中非常常见的问题。DMA 写了 DDR，CPU 读不到最新数据； CPU 写了数据，DMA 读到旧数据；这些问题往往不是硬件坏了，而是 Cache 维护策略不正确。

### 4. 地址映射规划不清晰

地址空间一旦被软件、Boot、调试工具、驱动和文档依赖，后期修改成本很高。

所以地址映射必须早期规划，并保留扩展空间。

### 5. 忽略存储安全

哪些地址可执行？哪些区域安全世界可访问？哪些区域普通外设不能 DMA？

密钥是否能被 CPU 直接读出？调试口是否能访问安全存储？这些都不是软件后期能完全补救的问题，必须在 SoC 架构阶段考虑。

可以把 SoC 存储子系统的设计原则总结成五句话。

第一，存储是分层的，不是一块大的内存。

不同数据应该放在不同速度、容量、成本和功耗特性的存储中。

第二，热数据靠近计算，冷数据放到远处。

寄存器、Cache、SRAM、DDR、Flash 形成从快到慢、从小到大的层级。

第三，Cache 提升平均性能，SPM 提升确定性。

通用计算依赖 Cache，实时控制和 DSP 场景常需要 SPM。

第四，DDR 是共享资源，不是无限带宽。

必须结合互联仲裁、QoS、DMA、双缓冲和带宽预算一起设计。

第五，地址映射是软硬件契约。

它影响启动、驱动、安全、调试和系统扩展，必须早期规划清楚。

SoC 的所有计算，最后都绕不开数据。数据从哪里来？放在哪里？谁能访问？访问多快？是否一致？掉电后是否还在？异常时能否恢复？安全数据能否被保护？这些问题共同构成了存储子系统设计。一个优秀的存储子系统，不是简单追求"更大的 DDR"或"更多的 SRAM"，而是让不同数据在正确的位置、以正确的方式被访问。它让 CPU 取指更快，让 DMA 搬运更顺，让 NPU 不被喂数拖慢，让显示不被带宽抖动打断，让 Boot 能可靠启动，让安全配置不可篡改，让系统在性能、成本、功耗和可靠性之间取得平衡。所以，存储子系统不是 SoC 的后台仓库。它是整个芯片的数据生命线。
