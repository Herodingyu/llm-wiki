# 只读、只写、读写寄存器的设计意图与避坑清单

> 来源：微信公众号「OneChan」
> 原文链接：https://mp.weixin.qq.com/s/-bDz9zU0869bCnlqttuFuw
> 记录时间：2026-05-04

---

## 核心观点

寄存器不是硬件给软件出的考试题，而是**硬件和软件之间的通信协议**。每一个寄存器的设计、每一个位的定义、每一种访问类型的选择，都不是随便拍脑袋决定的，背后都有硬件工程师的深思熟虑。

## 只读寄存器（RO）：不是为了不让你写，而是为了保护你

### 设计意图
只读寄存器的本质是**硬件给软件的一个"安全承诺"**——保证这个寄存器的值永远不会被软件意外修改。

### 案例：英飞凌 TC3xx SMU 故障状态寄存器
- 全部为硬件级只读，即使 CPU 处于最高特权级也写不了
- 如果可写，软件 bug 可能意外清除故障状态，导致安全机制未触发
- 汽车 120km/h 行驶时刹车系统故障被清除，后果不堪设想

### 冷知识
**99% 的只读寄存器，在硬件内部其实是可写的**，只是硬件只允许自己写，不允许软件写。例如 CPU 的程序计数器 PC，软件看来是只读的，但硬件内部每时每刻都在修改。

### 避坑清单

| 坑点编号 | 常见错误 | 正确做法 | 车规要求 |
|---------|---------|---------|---------|
| R-01 | 向只读寄存器写入任何值 | 永远不要对只读寄存器执行写操作 | ASIL-B 及以上：静态代码检查中禁止 |
| R-02 | 依赖只读寄存器上电默认值 | 上电后必须显式读取确认初始状态 | ASIL-D：关键状态寄存器初始值校验 |
| R-03 | 中断上下文频繁读取只读寄存器 | 缓存在 RAM 中定期更新 | 中断服务程序执行时间 ≤1μs |
| R-04 | 认为只读寄存器值不会被硬件异步修改 | 每次使用前都必须重新读取 | 禁止缓存关键状态寄存器超过 1ms |

## 只写寄存器（WO）：最反人类的设计，却是最聪明的设计

### 设计意图
**从根本上消除"读-修改-写"问题**。

### 案例：GPIO 输出寄存器
软件想把第 5 个引脚设置为高电平，其他保持不变：

```c
// 读写寄存器：需要读-修改-写，存在竞态风险
GPIO->PDOR |= (1 << 5);  // 读 → 改 → 写，三步非原子

// 只写寄存器：一步完成，无竞态风险
GPIO->PSOR = (1 << 5);   // 写 1 置位，写 0 忽略
```

**NXP S32K 车规芯片设计**：
- `PSOR`：置位寄存器，写 1 置位，写 0 忽略
- `PCOR`：清零寄存器，写 1 清零，写 0 忽略
- `PTOR`：翻转寄存器，写 1 翻转，写 0 忽略

### 避坑清单

| 坑点编号 | 常见错误 | 正确做法 | 车规要求 |
|---------|---------|---------|---------|
| W-01 | 尝试读取只写寄存器校验写入结果 | 永远不要对只写寄存器执行读操作 | 静态代码检查中禁止 |
| W-02 | 对只写寄存器使用"读-修改-写" | 永远使用单次写操作 | ASIL-C 及以上：禁止使用 RMW |
| W-03 | 忘记"写 1 生效，写 0 忽略" | 只向需要修改的位写 1，其他位写 0 | 必须使用宏定义，禁止直接写数值 |
| W-04 | 多个任务同时操作同一个只写寄存器 | 使用互斥锁保护 | 共享只写寄存器必须使用互斥锁 |

**大厂标准宏定义示例**：
```c
// 正确
#define GPIO_SET_PIN(pin)   (GPIO->PSOR = (1U << (pin)))
#define GPIO_CLEAR_PIN(pin) (GPIO->PCOR = (1U << (pin)))
#define GPIO_TOGGLE_PIN(pin)(GPIO->PTOR = (1U << (pin)))

// 错误：绝对禁止
#define GPIO_SET_PIN(pin)   (GPIO->PDOR |= (1U << (pin)))
```

## 读写寄存器（RW）：看起来最简单，其实坑最多

### 坑 1：写 1 清 0 的寄存器
```c
// 错误：会清掉所有已经挂起的中断
uint32_t pend = INT->PEND;
pend &= ~(1 << 5);
INT->PEND = pend;

// 正确：只清掉第 5 位中断
INT->PEND = (1 << 5);
```

### 坑 2：写操作和读操作指向不同的物理寄存器
典型例子是 SPI 的数据寄存器：
- 写 → 发送 FIFO（TDR）
- 读 → 接收 FIFO（RDR）

写进去一个值马上读出来，读到的不是写入的值，而是之前接收到的数据。这不是 bug，这就是设计。

### 坑 3：写操作触发硬件动作，读操作返回动作结果
ADC 转换寄存器：
- 写 → 触发一次 ADC 转换
- 读 → 读到转换结果

写入值和读出值完全无关。

### 坑 4：写操作有延迟，读操作返回旧值
CAN 控制器波特率配置寄存器：写完后必须等待硬件完成配置，才能进行下一步操作。写完马上读会得到旧值。

### 避坑清单

| 坑点编号 | 常见错误 | 正确做法 | 车规要求 |
|---------|---------|---------|---------|
| RW-01 | 对"写 1 清 0"寄存器使用读-改-写 | 直接写入需要清零的位掩码 | ASIL-B 及以上：必须经过代码评审 |
| RW-02 | 对"读写分离"寄存器执行写后读校验 | 不要对读写分离寄存器执行写后读校验 | 禁止对 SPI/I2C/CAN 数据寄存器写后读 |
| RW-03 | 认为写操作会立即生效 | 写完有延迟的寄存器后必须等待硬件确认 | 必须使用"配置完成"标志位确认 |
| RW-04 | 向保留位写入非 0 值 | 所有保留位必须写入 0 | ASIL-D：必须显式清零保留位 |
| RW-05 | 对"自清零"位执行写操作 | 不要对自清零位执行任何写操作 | 禁止对自清零位执行写操作 |

## 通用避坑原则

1. **永远不要相信你的直觉** — 行为不符合预期时，先看手册"寄存器描述"部分
2. **永远使用宏定义操作寄存器** — 不要直接写数值
3. **永远不要在中断上下文中操作复杂的寄存器**
4. **永远不要假设寄存器的默认值** — 上电后必须显式配置
5. **永远保留寄存器操作的日志** — 调试版本记录所有读写操作

## 车规级特殊要求

- 所有寄存器操作必须是**原子操作**，禁止被中断打断
- 关键寄存器配置必须进行**双写校验**：写入 → 读出确认 → 不匹配则重新写入
- 所有保留位必须**显式写入 0**
- 禁止使用**位带操作**访问安全相关寄存器
- 所有寄存器操作的时间必须**可预测**
