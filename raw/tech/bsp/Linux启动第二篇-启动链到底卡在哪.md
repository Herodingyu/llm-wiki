很多人第一次做嵌入式 Linux，会有一个很自然的误解：

板子都上电了，串口也亮了，系统不就应该起来吗？

但真正做过板级调试的人都知道，事实远没有这么简单。

因为"板子上电"只代表一件事：

**硬件开始工作了。**

而"系统起来"意味着另一件事：

**整条启动链每一环都正确接力了。**

只要中间有一环出问题，系统就可能停住：

• 可能连 U-Boot 都看不到
• 可能停在 U-Boot 命令行
• 可能内核已经开始打印，却最后 panic
• 也可能 rootfs 挂不上，永远进不了用户空间

所以，想搞懂"为什么板子上电后系统却起不来"，关键不是盯着某一行日志，而是先回答一个问题：

**它到底卡在启动链的哪一层？**

## 一句话先讲明白

如果只记一句话，可以先记这个：

板子上电不等于系统启动，Linux 启动本质上是一场分层接力：BootROM、Bootloader、Kernel、rootfs、userspace，任何一层掉棒，系统都起不来。

所以现场排查时，最重要的不是"一顿改配置"，而是先判断：

• 是不是连第一阶段都没走通
• 是不是卡在 bootloader
• 是不是内核没起来
• 是不是 rootfs 挂不上
• 还是用户空间根本没跑起来

## 第一层：卡在 BootROM 或更早阶段

如果板子上电后：

• 串口完全没输出
• 没有任何启动痕迹
• 连 U-Boot banner 都看不到

那第一反应就不该是 Linux，而应先怀疑最早阶段。

这一层最常见的问题包括：

• 启动介质没识别到
• 启动拨码或 strap 配置错误
• 烧录镜像格式不对
• 供电、时钟异常
• DDR 初始化前就已经出问题
• BootROM 根本没找到合法镜像

这一层的特点是：

**系统还没进入你熟悉的软件世界。**

所以这时不要先看 bootargs、rootfs，也不要先怀疑内核，而要先确认：

• 芯片到底从哪里启动
• 启动介质上有没有正确镜像
• BootROM 能不能把第一阶段引导程序拉起来

很多"系统起不来"的问题，实际上连 Linux 这一步都还没碰到。

## 第二层：卡在 Bootloader

如果串口已经能看到：

```
SPL
TF-A
U-Boot banner
Hit any key to stop autoboot
=> 命令行
```

说明前一阶段至少已经走通了，系统已经进入 Bootloader 世界。

这时最常见的问题包括：

• bootcmd 配错
• bootdelay 或 preboot 异常
• 启动介质选错
• kernel / dtb 没加载成功
• bootm、booti、bootz 用错
• 网络启动、MMC 启动链路有问题

这一层有一个很典型的现象：

**板子不是死机，而是停在 U-Boot。**

比如：

• 自动启动没跑起来
• mmc read 报错
• tftpboot 失败
• Wrong Image Format
• Bad Linux magic

看到这类问题时，重点应该放在：

• 镜像是否存在
• 地址是否正确
• 启动命令与镜像格式是否匹配
• 设备树是否加载成功

也就是说，这时问题通常不是 Linux 本身，而是：

**Bootloader 没把 Linux 正确交出去。**

## 第三层：内核起来了，但没完全起来

如果你已经看到 Linux 内核日志了，比如：

```
Starting kernel ...
early boot log
CPU、memory、driver 初始化日志
```

那说明控制权已经交给内核了。

这时如果系统 still 起不来，常见问题包括：

• 串口参数不对，看起来像"没日志"
• earlycon 没配，早期日志看不到
• 驱动初始化卡住
• 内核访问异常导致 oops
• 致命错误导致 panic
• watchdog 最后把系统复位

这一层的特点是：

**Linux 已经开始跑了，但还没站稳。**

这时候最重要的不是只看最后一行，而是看前面有没有这些关键词：

```
oops
panic
Call trace
Unable to handle kernel NULL pointer dereference
watchdog
```

如果看到这些，重点就该切到：

• 驱动初始化
• 内核配置
• 设备树资源
• 串口/console 参数
• 早期启动路径

## 第四层：内核起来了，但 rootfs 挂不上

这是现场特别高频的一类问题。

最经典的报错就是：

```
VFS: Cannot open root device
Waiting for root device
Kernel panic - not syncing: VFS: Unable to mount root fs
```

这类问题很容易被误判成"内核坏了"，其实很多时候并不是。

它真正说明的是：

**内核已经起来了，但找不到根文件系统。**

这一层最常见的原因有：

• bootargs 里的 root= 写错
• SD 和 eMMC 设备号搞反
• 分区号写错
• 忘了加 rootwait
• 存储驱动没起来
• 设备树里存储控制器配置不对
• 文件系统驱动没编进内核

所以看到 VFS 报错时，重点不要先放在 U-Boot，而要先看：

• root= 到底写了什么
• 块设备有没有枚举出来
• rootfs 所在介质驱动是否正常
• 分区是否真实存在

这类问题的本质不是"Linux 没起来"，而是：

**Linux 起了，但没有家。**

## 第五层：rootfs 挂上了，但系统还是进不去

还有一种情况是，内核没报 VFS，rootfs 也挂上了，但系统依然起不来。

这时要怀疑的是最后一层：

**用户空间没有真正跑起来。**

比如：

• /sbin/init 不存在
• init 程序损坏
• rootfs 不完整
• 用户空间关键脚本异常
• systemd / busybox init 启动失败

这一层的特点是：

• 内核日志基本正常
• rootfs 看起来也没问题
• 但系统迟迟进不了 shell 或业务态

这说明问题已经不在"内核是否启动"，而在"用户空间是否接住最后一棒"。

## 现场最实用的判断方法

以后再遇到"板子上电后系统起不来"，建议先按这四步判断：

**第一步：先看有没有任何输出**

没有输出，优先怀疑 BootROM、介质、供电、时钟、早期引导。

**第二步：看能不能进 U-Boot**

能进 U-Boot，说明前一阶段没大问题，重点转到镜像加载和启动命令。

**第三步：看内核日志有没有出来**

如果 Starting kernel 之后没东西，重点看 console、earlycon、内核早期路径。

**第四步：看有没有 VFS、panic、watchdog**

• VFS：重点查 rootfs
• panic/oops：重点查内核和驱动
• watchdog：重点查前面是不是卡死了

你会发现，一旦按层拆，启动问题就没那么乱。

## 结尾

板子上电，只说明系统开始尝试启动；

真正"系统起来"，意味着整条启动链都走通了：

• BootROM 找到入口
• Bootloader 把内核拉起来
• 内核完成初始化
• rootfs 正常挂载
• 用户空间真正运行

所以以后再遇到启动失败，不要一上来就乱改镜像、乱改参数。

先问自己一句：

**它到底卡在启动链的哪一层？**

只要这个问题先想清楚，排查效率会高很多。
