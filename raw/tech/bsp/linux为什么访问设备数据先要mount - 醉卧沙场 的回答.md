## linux为什么访问设备数据先要mount?

在知乎上，我几乎一直在不厌其烦的强调“ 存储设备 ”和“文件系统”是两个不同的概念。 /dev/下的存储设备文件对应的是存储设备本身，你虽然也可以打开、读取、写入一个存储设备，但是你面向的终究是一个存储设备，不是文件系统。存储设备提供的访问单元是块（比如扇区），对于一个 块设备 来说，它能够提供的基本功能就是可以寻址并读/写哪个/组地址“块”。关于编址可以参考我下面这个回答： 比如你可以试图访问某一个或几个扇区的数据，把扇区里的数据读出来，读出来一堆0和1，但是对于一个庞大的存储设备，你很难知道哪个块里是什么数据，更难以理解读出的一块数据是什么意思。 所以“块”作为计算机面向的单位还是可以的，但绝不适合作为用户面向的单位。用户需要面向的单位不是存储块本身，用户面向的单位是“文件”这样的抽象概念，而文件这个概念就是文件系统抽象出来的，一个文件的数据（和元数据）可能散落在一个存储设备的各个角落（块），用户如果想通过直接读取存储块的内容的方式来获取文件内容是非常困难的，和大海捞针一样。所以用户需要文件系统提供的一层逻辑抽象，借助文件系统的这层管理逻辑来以“文件”的概念访问存储设备。 那么如何借助文件系统来访问存储设备呢？这时就到我们的 “挂载”也就是mount操作出场的时候了。挂载的最主要作用其实就是将一个具体存储设备上的具体文件系统和操作系统中对应的 文件系统驱动 （/模块）关联起来，并将这个具体文件系统中的文件和目录关系挂载到全局目录树上，形成一个“激活运行状态”的文件系统 。这之后用户才可以看到你已经习以为常、甚至认为理所当然存在的那些“文件”和“文件夹”，并能通过文件系统模块提供的很多方法访问这些“抽象”概念。 文件系统是一个说大不大说小不小的概念，我不可能在一篇回答里就把它说清楚。好在我之前还回答过很多这方面的问题，如果你真的对文件系统的组织/运行逻辑感兴趣，可以参考下面几篇回答/文章（第4个链接和这个问题很相关，其它三个可以为你提供更多相关方面的理解）： 如果还想了解更多，可以参考下面这篇索引式文章的“存储和文件系统”部分列出的文章和回答： 当然，如果你想认真的研究和学习，那这些都是不够的，还需要系统的学习计算机知识，并不断通过研究和实践来认识文件系统和操作系统。 补充: 忽然觉得好多人还是难以理解存储介质和文件系统的关系和区别, 当然这也没什么, 术业有专攻, 我们不能要求所有人都是计算机专业的。普通的电子设备用户确实没有必要做到能将硬件介质和软件逻辑特别明确分开的地步, 大家常用的Windows系统一直以来也都在尽量透明化存储分区和文件系统的概念, 用户一般口头一提就是C盘, D盘这样的盘符。 脑子里想的应该也是"叫C盘的分区, 叫G盘光驱, 叫H盘的U盘"等等这样的东西, 根本不会想上面的文件系统, 更不会想到什么挂载的概念(被windows透明化了)。其实你双击这些盘符时打开的不是存储设备本身，而是已经挂载的存储设备上的文件系统的挂载点。对普通用户透明化很多细节概念其实是没有问题的, 这应该是属于针对用户使用而设计的概念, 和计算机原理没太大关系。 为了让非计算机专业的普通用户能形象化的理解存储介质和文件系统的区别和联系, 我举一个类比的例子(我也将这个例子回复给了评论区的一个朋友): 分区也好，硬盘也罢，甚至冗余存储设备、 逻辑卷 ，又或者通过 iscsi 、 FC链接 的存储设备，这些都是存储设备而已，就好像你有一个空白的本子，不管是牛皮纸的也好，镶金边的也好，活页本也好，甚至是word文档或者能在线编辑的文档，这些都是不同形式的存储介质而已。 在写入有意义的文字之前它们都是“可以涂写的介质”而已，但是比如你写入了小说，加上了插画，你就获得了句子、段落、章节、主题、日期、主人公、叙事手法、经典句子等等这些概念。你在和别人讨论时你会说哪一节写的怎么样，哪个人物在哪一段怎么样，但是你一般不会说第几页从上往下数10厘米，再从左往右数3到8厘米那个内容写的特别好。再比如你画了一幅画, 你会获得近景, 远景, 线条, 人物, 花草树木, 天空, 房子, 色彩, 冷暖色等等这些概念(非艺术专业,我就随便举个例子:) 这些概念本身都是超脱存储介质(如纸张, 画本)的存在，但是又经常依附于书本这样的存储介质而呈现。这就好比文件系统和存储介质的关系了，请类比的理解。最后我只能再贴一个我觉得最浅的概述性回答了： 我感觉这应该是我能说的最通俗的解释了, 不想明白这些的普通用户大可不必过于纠结, 而非要纠结的想弄明白的, 那我只能推荐你系统的学习计算机系统的基础知识了。倒不用太深的知识, 基本上到计算机专业大二下学期或大三上学期的专业基础知识可能就差不多了.

题主需要了解一下 **文件系统** 这个概念。

理论上是可以直接读取/dev/sdb4里的内容的，比如用下面的命令可以把/dev/sdb4里的前8M长的内容原封不动地复制到/tmp/test这个文件里。

```
# dd if=/dev/sdb4 of=/tmp/test bs=4M count=2
```

但是你会发现，复制出来的内容是看似杂乱无章的二进制数据，并不能读取有效的文件内容。

U盘（或者任何存储设备）里，除了存储的文件本身以外，还有各种元数据，比如文件名，路径，权限等信息。这些元数据通过文件系统组织起来，才能让用户“有秩序地”访问文件。Linux上主流的文件系统是ext4, Windows上主流的文件系统是NTFS.

mount不仅仅是在目录树上挂在一个节点，还做了很多其他操作。比如自动或手动识别设备里的文件系统，调用适当的驱动去读取文件系统的元数据，还有一些其他的初始化操作。执行了这些操作以后，你才能通过ls命令看到U盘里的文件夹、文件。

linux系统的设计传承unix系统设计的很多基本思路，其中一个就是万物皆文件的概念。

注意这里的“文件”是广义的。是一个可以对其进行打开、关闭、读取和写入数据的实体单位。

这个实体单位可以是逻辑意义上的，比如大家熟知的pdf文件、图片、视频或者音乐。

这个实体单位也可以是物理意义上的，比如CPU、GPU、内存、磁盘、网卡、打印机等等。

甚至诸如进程、线程、内核对象（锁、信号量等等）均可看作文件。

它们共通的特点都是你可以对其进行打开和关闭，可以从其获得信息，也可以将信息写入其中。

所以它们都被抽象成为文件，这一概念。

/dev/目录下的设备，就是被称为设备文件的这样一群文件。它们通常对应某个设备，当然不仅仅是物理设备，还可以包括一些软件定义的虚拟逻辑设备。

（另外，在\*nix系统当中，文件夹其实也是一种文件）

当然另外一方面，这些不同类型的文件，其内部的信息格式，以及与其交互的具体方式，是千差万别的。

因此为了使用它们，我们必须为每种文件类型编写专门的接口程序。这种接口程序，就被称为“驱动”。当然，对于诸如文档、图片、视频等文件我们习惯上更加喜欢称这种接口程序为解析器或者编码器，或者文件格式插件。但是本质上它们是类似的东西。

当系统加电启动之后，系统侦测与其相连接的各种设备，并将其作为文件登记在/dev目录之下。这些文件代表的是这些设备本身，而这些文件的解析器/编码器，实际上就是这些设备的驱动程序。

此时我们如果使用诸如open/close/read/write等系统接口对这些设备文件进行操作，我们就可以对其对应的设备进行控制和信息交互了。

但是，对于一些设备，如硬盘等，为了能够更方便地使用其海量的存储空间，我们往往会对其的使用附加一套另外的规则。这就好像我们学校里的宿舍楼，原本它们都只是楼，你可以入住任何一间。但是在实际分配的时候，出于管理上的需要，总会根据专业年级性别等对可以入住的区域进行划分。

这种额外的规则，其本身并不是设备的一部分。而是依附在具体设备上，但又独立存在的逻辑概念。

对于计算机系统当中的存储设备，这种对于存储区域的预划分以及相应的规则集合体，被称为文件系统。因为是完全的逻辑概念，在实际运用当中，它总是依附具体的设备存在。但是其本身并不是设备的一部分。

所以在\*nix系统当中，设备和文件系统被映射成为了不同的文件实体。/dev下的为设备实体，其对应的“驱动”虽然知道如何和设备本身通信，却对设备上数据的实际组织形式，（在上面这个硬盘的例子当中）也就是文件系统一无所知。

这就好像我们去某个学校找某位同学，地图虽然能告诉你学校宿舍在哪里，却无法告诉你你要找的同学在哪里。因为它没有宿舍分配的信息。

所以，为了能够找到我们想要的数据，我们需要对这个/dev下的这个设备使用mount。mount所做的事情其实就是通过一些方法去推测设备上数据组织的形式（也就是文件系统的类型），然后找到对应的“驱动”（文件系统解析/编码程序），与挂载点相关联。

这样当你访问挂载点时，系统就知道该用哪种数据组织形式去和哪个设备通信。这就是mount的作用。

如果理解了这些，那么应该很快会想到，如果我们知道如何去编码/解码设备返回的数据流，那么其实不mount也是可以直接通过/dev去访问设备的。这就是所谓的raw格式。一些数据库软件为了提高性能，有的时候会使用这种方法。

至于其他回答提到的block（块）设备，我觉得和这个问题本身没有什么必然的关系。当然一般来说文件系统大多是为了解决随机读写，而随机读写一般都是块设备。但是这应该不是必然的。
