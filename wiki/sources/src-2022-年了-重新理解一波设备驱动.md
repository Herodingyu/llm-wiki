## Summary

本文从工程实践角度重新梳理了Linux设备驱动的核心概念，强调在真实工作场景中应避免从零构建驱动，而应在内核现有的模块化驱动框架基础上扩展。文章以字符设备驱动为例，详细讲解了设备号的分配、文件操作接口的实现（open/read/write/ioctl）、以及cdev注册流程。并通过LED驱动实例对比了"从零编写"与"基于框架扩展"两种方式的差异，展示了如何利用内核提供的LED子系统、platform驱动框架、pinctrl/gpio子系统等现有基础设施，以更少代码实现更规范、更可维护的驱动程序。

## Key Points

### 1. 设备驱动的本质
- **定义**：设备驱动程序是对硬件的抽象，为应用程序提供统一访问接口
- **运行空间**：通常运行在内核空间（也可通过UIO/I2CDEV等接口在用户空间运行）
- **字符设备**：最常见的硬件抽象，/dev目录下的设备节点文件是内核导出的访问接口
- **设备节点三要素**：Type（block/char）、Major number（设备类）、Minor number（具体设备）

### 2. 从零编写字符设备驱动
- **步骤1**：分配设备号（`register_chrdev_region()` 或 `alloc_chrdev_region()`）
- **步骤2**：实现文件操作（open、read、write、ioctl）
- **步骤3**：注册字符设备（`cdev_init()` + `cdev_add()`）
- **缺点**：代码冗余、缺乏标准化、难以维护

### 3. 基于框架的驱动开发（推荐）
- **LED子系统**：利用内核`led_classdev`框架，只需实现brightness_set接口
- **Platform驱动框架**：自动处理设备树匹配、资源获取、probe/remove生命周期
- **Pinctrl子系统**：统一管理引脚复用和电气属性
- **GPIO子系统**：标准化GPIO操作接口
- **优点**：代码量少、可维护性高、与内核生态兼容

### 4. 工程实践建议
- 避免从零构建驱动，优先复用内核现有框架
- 使用设备树（Device Tree）描述硬件，实现驱动与硬件解耦
- 利用sysfs/debugfs提供调试接口
- 遵循内核编码风格和接口规范

## Key Quotes

> "多年来，我接触到的 Linux 驱动教程大多都是从 0 编写，这样在真正的工作场景里，其实是应该尽量避免从 0 构建自己的设备驱动的。"

> "更好的做法是在高度模块化的驱动框架里添加自己的设备驱动。这样做的好处是最大程度地复用内核现有的代码，同时获得极大的弹性和可维护性。"

> "设备驱动程序 (device driver) 是对硬件的抽象。"
