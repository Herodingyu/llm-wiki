# 从上电到内核：瑞芯微 RK3568/RK3588 启动时序，一篇讲透

做嵌入式这行，几乎每个人都有过这样的时刻：板子接上串口，电源一插，盯着屏幕等那一行行 log 滚出来。U-Boot 的版本号、DDR 的容量、Starting kernel ... 一旦卡在某一行不动，心里就开始打鼓——到底是哪一级挂了？

本文以 RK3568/RK3588 为例，把从上电到内核的启动链路从头到尾捋清楚：芯片刚上电时在哪跑代码、DDR 是怎么被拉起来的、镜像是怎么一级一级搬进内存的、ARM 的特权级又是怎么一层层交出去的，最后内核怎么接过控制权。

## 一、先看全局：五级启动链路

不管是 RK3568 还是 RK3588，启动的骨架是一样的，可以拆成五级：

| 阶段 | 名字 | 跑在哪 | 核心任务 |
| --- | --- | --- | --- |
| 0 | BootROM | 片内 SRAM | 上电复位，扫描启动介质，找到第一段可执行代码 |
| 1 | DDR 初始化 | 片内 SRAM | 把外部 DDR 内存拉起来，后面才有大块空间用 |
| 2 | 搬运器（SPL/Miniloader） | 进入 DDR | 把 ATF + U-Boot 从存储搬到 DDR |
| 3 | ATF + U-Boot | DDR | 安装安全服务、降特权级、加载内核和设备树 |
| 4 | Linux Kernel | DDR | 解压、解析 DTB、挂根文件系统、起用户空间 |

这条链路有两个贯穿始终的主线：
- **代码运行的位置在变**（从几十 KB 的片内 SRAM，搬到几个 GB 的外部 DDR）
- **CPU 的特权级也在变**（从最高的 EL3 一路降到用户态的 EL0）

## 二、第 0 级：BootROM——芯片睁眼第一件事

BootROM 是固化在芯片内部的一段只读代码，出厂就烧死在片内 ROM 里，谁也改不了。它是芯片上电复位后执行的第一段指令。

这时候外部 DDR 还没初始化，根本不能用。所以 BootROM 只能跑在 **片内 SRAM** 里——空间很小，也就几十 KB 到几百 KB，干不了什么重活。它的任务很专一：

- 按固定优先级扫描启动介质找第一段代码：
  - **正常上电（Pre-loader）**：`SD 卡 → SPI NOR → SPI NAND → eMMC`
  - **Maskrom 模式**：`SPI NOR → SPI NAND → eMMC → SD 卡`
- 在介质上找一个叫 **IDBlock** 的结构，靠魔数 `0x0FF0AA55` 识别
- 找到后，把第一段用户代码（DDR 初始化代码）加载进 SRAM 并跳过去执行

如果所有介质都扫不到合法的 IDBlock，BootROM 会进入 **MaskROM 模式**——这就是我们用 RKDevTool 烧录、或者板子刷砖后重新灌固件时进的那个模式。它本质上是 BootROM 走投无路后，老老实实等着上位机通过 USB 把代码喂进来。

> **实战提示**：很多"板子不启动、串口无任何输出"的问题，根子在这一级——要么 IDBlock 没烧对，要么启动介质的拨码/电阻选错了，BootROM 压根没找到第一段代码。

## 三、第 1 级：DDR 初始化——把内存拉起来

BootROM 加载进来的第一段代码，核心使命就一个：**初始化 DDR**。

DDR 这东西不是上电就能用的，得做一整套复杂的初始化——时序参数配置、读写训练（training）、阻抗校准等等，全部跑完，几个 GB 的内存才真正可用。这一步还是在片内 SRAM 里跑，因为 DDR 自己还没好。

到这里，瑞芯微就分出了 **两条链路**：

- **旧链路（rkbin 闭源方案）**：用 RK 预编译好的 `ddr.bin`。它放在 rkbin 仓库里，是个二进制 blob，不开源。好处是出厂兼容性好、稳定；坏处是黑盒，没法改。
- **新链路（U-Boot 全开源方案）**：用 U-Boot 的 **TPL**（Tiny Program Loader）来做 DDR 初始化，对应代码在：
  - `arch/arm/mach-rockchip/tpl.c`
  - `drivers/ram/rockchip/sdram_rkxx.c`（各芯片型号的 DDR 驱动）

TPL 是 U-Boot 体系里专门为"DDR 还没好、只能跑在 SRAM"这个场景准备的超精简阶段，编译出来非常小，刚好塞进 SRAM。

## 四、第 2 级：搬运器——把后面的镜像搬进 DDR

DDR 一旦就绪，下一步就是把后续的大块镜像（ATF + U-Boot）从存储介质搬到 DDR 里。承担这个活的，旧链路叫 **Miniloader**，新链路叫 **U-Boot SPL**。

按 RK 官方《U-Boot Next Dev 开发指南》的定义，这两者职责完全对应：

| 旧链路 | 新链路 | 说明 |
| --- | --- | --- |
| ddr.bin | TPL | DDR 初始化 |
| miniloader | SPL | 搬运器 |

- **SPL 运行在 DDR 里**（这点和只能跑 SRAM 的 TPL 不同，DDR 已经可用，空间宽裕），完成系统的 lowlevel 初始化
- 加载后续固件：把 **TRUST（trust.img，armv8 平台即 ATF bl31）** 和 **U-Boot proper（uboot.img）** 从 Flash/eMMC 读到 DDR 指定地址
- 跳转过去，把接力棒交给 TRUST（ATF）

对应到新链路的代码：`arch/arm/mach-rockchip/spl.c`

这一级是"代码搬家"最关键的一步：在它之前，所有东西都挤在巴掌大的 SRAM 里；在它之后，完整的运行环境在 DDR 里建立起来，后面的内核、文件系统才有地方放。

## 五、第 3 级：ATF + U-Boot——装好安全服务，再一层层下放权限

这一级是 ARMv8 体系最有讲究的地方，涉及**特权级（Exception Level）**的概念。

ARMv8 把 CPU 的运行权限分成四级：

- **EL3**：最高，安全监控（Secure Monitor），ATF 跑在这
- **EL2**：Hypervisor 级，U-Boot proper、虚拟化跑在这
- **EL1**：内核态，Linux Kernel 跑在这
- **EL0**：用户态，应用进程跑在这

启动流程在这一级要做两件事：

### 1）ATF（bl31）在 EL3 安家

ATF 全称 ARM Trusted Firmware，是 ARM 官方的可信固件。bl31 是它的运行时部分，跑在最高特权级 EL3。它的任务是安装一系列 **运行时服务**，最重要的是 **PSCI**（电源状态协调接口）和 **SMC** 调用入口。装好之后，它通过 `ERET` 指令把特权级 **降到 EL2**，把控制权交给 U-Boot。

对应代码路径：
- `arm-trusted-firmware/plat/rockchip/`（RK 各平台的 ATF 移植）
- `optee_os/core/arch/arm/plat-rockchip/`（如果带 OP-TEE 安全 OS）

### 2）U-Boot proper 在 EL2 干引导的活

U-Boot 这时候降到了 EL2，开始干我们最熟悉的事：

- 初始化各种外设（网口、USB、显示等）
- 从存储/网络加载内核镜像和设备树（DTB）
- 准备好引导参数（bootargs、内存布局等）
- 跳转到内核，特权级再降一级到 **EL1**

这里有个关键点：内核运行起来后，遇到 CPU 上下电、休眠唤醒这类电源管理操作，会通过 **SMC 指令陷回 EL3**，由 ATF 的 PSCI 服务处理。所以 ATF 不是装完就退场了，它在整个系统运行期间一直待在 EL3 当"守门人"。

## 六、第 4 级：Linux Kernel——接过控制权

U-Boot 跳转过来后，内核在 EL1 启动，流程大家比较熟了：

- **自解压**：内核镜像通常是压缩的（zImage/Image.gz），先解压到内存
- **解析设备树**：读 U-Boot 传过来的 DTB，搞清楚板子上有哪些硬件
- **初始化内核子系统**：内存管理、调度器、各种驱动依次起来
- **挂载根文件系统**：从 eMMC/SD/NFS 找到 rootfs 挂上
- **启动 init 进程**：第一个用户空间进程拉起，特权级降到 **EL0**，系统正式进入用户态

到这一步，从上电到用户空间的整条链路就走完了。

## 七、代码路径速查表

| 阶段 | 功能 | 代码路径（U-Boot 新链路） |
| --- | --- | --- |
| DDR 初始化 | TPL | `arch/arm/mach-rockchip/tpl.c` |
| DDR 驱动 | 各型号 SDRAM | `drivers/ram/rockchip/sdram_rkxx.c` |
| 搬运器 | SPL | `arch/arm/mach-rockchip/spl.c` |
| 安全固件 | ATF（bl31） | `arm-trusted-firmware/plat/rockchip/` |
| 安全 OS | OP-TEE（可选） | `optee_os/core/arch/arm/plat-rockchip/` |
| 打包 | 镜像合并 | `make.sh` / `mkimage`（新）、`boot_merger`（旧） |

旧链路的 `ddr.bin`、`miniloader`、`bl31.elf` 都来自 **rkbin** 仓库的预编译二进制，配套的 `*.ini` 文件描述各段的加载地址。

## 八、调试定位速查

搞清楚时序，最大的好处是 **定位问题能一刀切到具体的级**。

| 现象 | 可能卡住的级 | 排查方向 |
| --- | --- | --- |
| 串口完全无输出 | 第 0 级 BootROM | 烧录、拨码、启动介质选择 |
| 只有 DDR 相关 log 然后卡死/重启 | 第 1 级 DDR 初始化 | DDR 型号、时序参数、训练失败 |
| U-Boot 起来了但加载内核失败 | 第 3 级 U-Boot | 分区表、内核镜像地址、DTB 不匹配 |
| Starting kernel... 后没下文 | 第 4 级 Kernel 早期 | 设备树、内存地址、console 参数 |

---

> **核心原则**：内存不够就先把内存搞大，权限太高就一层层往下交，环境不全就一级一级往上搭。想通了这一点，不管换 RK 哪颗芯片，甚至换到别家的 ARMv8 平台，套路都是相通的。
