## Summary

本文从计算机系统原理角度深入解答了"为什么Linux访问设备数据需要先mount"的问题。文章强调存储设备和文件系统是两个不同的概念：/dev下的设备文件对应存储设备本身（提供块访问），而文件系统是在存储介质之上建立的逻辑抽象（提供文件/文件夹访问）。核心在于理解mount操作的真正作用——将具体存储设备上的具体文件系统与操作系统中的文件系统驱动关联，并将文件和目录关系挂载到全局目录树上，形成"激活运行状态"。没有mount，用户只能看到原始的二进制块数据，无法理解其中的文件结构。

## Key Points

### 1. 核心概念区分
- **存储设备**：物理介质（硬盘、U盘、SSD），提供块（扇区）访问
- **文件系统**：存储设备之上的逻辑抽象，提供文件/文件夹/元数据管理
- **设备文件**（/dev下）：代表存储设备本身，面向块操作

### 2. 为什么不能直接访问/dev？
- **可以读取原始数据**：`dd if=/dev/sdb4 of=/tmp/test bs=4M count=2`
- **但数据无意义**：复制出来的是杂乱无章的二进制，无法直接识别有效文件内容
- **缺少元数据**：文件名、路径、权限、目录结构等信息由文件系统管理
- **块 ≠ 文件**：存储设备提供块访问，用户需要的是文件抽象

### 3. Mount 的本质
- **关联设备与驱动**：将存储设备上的文件系统与内核文件系统驱动模块关联
- **激活文件系统**：形成"激活运行状态"，解析文件系统元数据
- **融入目录树**：将设备的文件和目录结构挂载到全局文件系统的某个节点
- **提供访问接口**：用户通过标准文件操作（open/read/write/ls）访问设备内容

### 4. 形象化类比
- **存储介质** = 空白本子（牛皮纸、活页本、Word文档）
- **文件系统** = 本子上的内容组织方式（章节、段落、主题、插画）
- **Mount** = 打开本子并理解其内容组织方式
- **直接读/dev** = 只拿到一叠空白纸，不知道上面写了什么

### 5. Windows 的透明化
- Windows自动完成mount过程（分配盘符）
- 用户感知的是"C盘""D盘"，而非底层mount操作
- 双击盘符时实际访问的是已挂载的文件系统挂载点

### 6. 驱动与文件系统
- **驱动**：为每种设备/文件类型编写的接口程序
- **/dev文件**：设备驱动程序与硬件交互的入口
- **Mount后**：文件系统驱动解析设备上的元数据，提供文件级访问

## Key Quotes

> "我几乎一直在不厌其烦的强调'存储设备'和'文件系统'是两个不同的概念。"

> "存储设备提供的访问单元是块，对于一个块设备来说，它能够提供的基本功能就是可以寻址并读/写哪个/组地址'块'。"

> "挂载的最主要作用其实就是将一个具体存储设备上的具体文件系统和操作系统中对应的文件系统驱动关联起来。"

> "分区也好，硬盘也罢，这些都是存储设备而已，就好像你有一个空白的本子。"
