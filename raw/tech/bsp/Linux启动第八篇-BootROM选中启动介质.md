很多人第一次看启动链时，会下意识觉得：

**系统能不能启动，主要看 U-Boot。**

因为它看得见、能进命令行、能改环境变量、还能直接做实验。

但如果按真正的启动顺序往前推，

决定系统"最开始从哪跑"的，其实不是 U-Boot，而是 BootROM。

也正因为这样，很多看起来像是"镜像烧录问题""U-Boot 问题""介质问题"的故障，

根子其实都在更前面那一层：

**BootROM 到底有没有按预期选中启动介质，并找到第一阶段镜像。**

如果只用一句话先讲明白这篇文章，那就是：

**BootROM 决定"从哪种介质起跑"，后面的 bootloader 决定"怎么继续往下跑"。**

这两件事，一定不要混。

## 一句话先讲明白

如果只记一句话，可以先记这个：

**SD、eMMC、SPI NOR、NAND 的区别，不只是存储介质不同，而是 BootROM 读取它们的规则完全不同。**

也就是说，

启动介质一变，BootROM 的读取方式、镜像位置、镜像格式要求，往往也跟着变。

所以很多时候不是"镜像坏了"，

而是：

**BootROM 没在它认得的地方，用它认得的规则，找到合法镜像。**

## 第一，BootROM 是怎么选启动介质的？

很多人以为 BootROM 会去"扫描一下系统里有哪些存储器，再挑一个能启动的"。

实际不是这样。

BootROM 的选择方式，通常更硬件化，也更死板。

它一般先读的是这些信息：

• strap 管脚
• 拨码开关
• eFuse / OTP
• 固定启动模式配置
• 芯片内部启动寄存器状态

也就是说，

系统到底从 SD、eMMC、SPI NOR、NAND，还是 USB 下载模式启动，

通常是 SoC 上电瞬间就已经被硬件配置决定了。

所以 BootROM 不是"运行时自由选择"，

而是"按芯片出厂定义的规则选启动路径"。

**启动介质首先是硬件定义的，不是 U-Boot 决定的。**

## 第二，BootROM 选中的，不只是"设备名字"

表面看，BootROM 像是在几种介质里选一个：

• SD
• eMMC
• SPI NOR
• NAND

但它真正选中的，其实是一整套读取规则。

因为这四种介质，对 BootROM 来说根本不是一回事。

它要同时确定的是：

• 通过哪个控制器访问
• 按什么协议读
• 从什么位置开始找镜像
• 镜像头格式怎么校验
• 第一阶段代码应该放到哪里执行

所以你看到的只是"换了个介质"，

但在 BootROM 看来，变掉的是**整条最早期加载路径**。

**启动介质不同，等于第一跳的游戏规则不同。**

## 第三，SD 启动为什么最常见？

因为它最适合开发调试。

BootROM 从 SD 启动时，通常会做几件很直接的事：

1. 初始化 SD/MMC 控制器
2. 识别卡是否存在
3. 从固定扇区或固定位置读取镜像
4. 检查镜像头是否合法
5. 把镜像搬到片内可运行内存
6. 跳转执行

这里最容易让人误解的一点是：

很多人会觉得，"我把文件拷到 SD 卡里了，系统为什么还不启动？"

原因是：

**BootROM 通常并不认识文件系统。**

它大多数时候不是去找某个文件名，

而是**直接按固定 offset 去读裸数据**。

所以 SD 启动的关键不是"文件在不在"，

而是：

• 镜像是不是放在 BootROM 规定的位置
• 格式是不是 BootROM 规定的格式

SD 启动之所以常用，不是因为它更高级，

而是因为它**最容易试错、最容易重刷、最适合 bring-up**。

## 第四，eMMC 为什么比 SD 更容易踩坑？

很多人会把 eMMC 理解成"板载 SD"。

入门时这么理解可以，但放到启动链里就不够了。

因为 eMMC 往往不只是一个简单存储区。

它通常还包含：

• User Area
• Boot Partition 1
• Boot Partition 2
• RPMB

这意味着对 BootROM 来说，

eMMC 不是"一个地方"，而是"多个可能的启动区域"。

所以 eMMC 启动经常会碰到这种情况：

• 镜像内容没问题
• 但写到了 user area，起不来
• 改写到 boot0，反而好了

这不是镜像本身变了，

而是 **BootROM 默认读取的区域变了**。

eMMC 启动常见问题一般集中在：

• boot partition 没使能
• 镜像写错区域
• 启动偏移不对
• 设备状态配置不对

**eMMC 启动最典型的坑，不是没烧进去，而是烧进去了但没烧到 BootROM 会去读的地方。**

## 第五，SPI NOR 为什么经常最稳？

因为它最简单。

BootROM 从 SPI NOR 启动时，整体逻辑通常非常直接：

1. 初始化 SPI 控制器
2. 发读命令
3. 从固定地址取镜像
4. 搬到 SRAM / OCRAM
5. 跳转执行

SPI NOR 对启动早期很友好，原因就在于：

1. 地址线性
2. 没有坏块管理
3. 不需要复杂 ECC
4. 读取模型简单
5. 很适合放第一阶段代码

所以很多系统都会这么设计：

• **SPI NOR 放 SPL / TF-A / U-Boot**
• **eMMC 或 NAND 放 kernel / rootfs**

因为最早期那一小段代码，最怕复杂性，最需要确定性。

**SPI NOR 启动的优势，不是容量大，而是第一跳足够简单。**

## 第六，NAND 为什么最容易让人困惑？

因为 NAND 把复杂性带到了启动最前面。

它和前面几种介质最大的不同在于：

• 按 page / block 组织
• 需要 ECC
• 可能有坏块
• 不能像 NOR 一样线性直接读

这就意味着，

如果 BootROM 要支持 NAND 启动，它自己必须内建一部分 NAND 读取规则，比如：

• page read
• ECC 校验方式
• 坏块跳过策略
• 固定镜像布局

而 BootROM 的能力通常是很有限的。

它不可能像完整 NAND 驱动一样处理各种复杂情况。

所以 NAND 启动最容易翻车的地方就在这里：

• 镜像确实烧进去了
• 但不符合 BootROM 预期的 page / block / ECC 布局
• 结果就是完全起不来

**NAND 启动最难的地方，不是写进去，而是按 BootROM 能理解的方式写进去。**

## 第七，为什么同一个镜像，换个介质就可能起不来？

这是最核心的问题。

因为"同一个镜像"这句话，很多时候只是在你看来内容一样。

但对 BootROM 来说，启动成功不只看内容，还要看：

• 放在什么位置
• 用什么格式封装
• 通过什么方式读取
• 当前介质是否满足它的启动规则

常见原因通常有这几类：

**镜像偏移不同**

SD、eMMC、SPI NOR、NAND 往往要求的起始位置不一样

**镜像头格式不同**

有些 SoC 需要特定 header，不是裸 bin 就能直接跑

**启动区域不同**

eMMC 可能要写 boot0，不是 user area

**ECC / 坏块规则不同**

尤其是 NAND，布局不对就等于没写对

**安全启动校验不同**

签名、认证头、打包方式都可能影响 BootROM 判断

所以很多时候真正的问题不是：

"这份镜像能不能启动？"

而是：

**这份镜像能不能按当前介质的规则，被 BootROM 正确识别并拉起来。**

## 第八，工程上怎么快速理解这四类启动介质？

你可以直接这么记：

**SD / eMMC**

本质上是块设备启动。

重点看扇区、偏移、启动分区、镜像写入位置。

**SPI NOR**

本质上是线性地址读取。

重点看固定地址、镜像头、第一阶段代码布局。

**NAND**

本质上是 page / block 启动。

重点看 ECC、坏块、OOB、镜像布局规则。

所以它们的差异，不只是"介质不同"，而是：

**BootROM 读取它们的方式，从底层上就不是同一种逻辑。**

## 第九，现场最容易忽略的判断点是什么？

最容易忽略的一点是：

**没有看到 U-Boot，并不一定是 U-Boot 出了问题。**

很多时候更前面的判断应该是：

• BootROM 有没有选到正确启动介质
• 这个介质有没有被正确识别
• 第一阶段镜像是不是放在对的位置
• 格式是不是 BootROM 能认的格式

因为一旦问题卡在 BootROM 这一层，常见现象往往就是：

• 上电后没输出
• 串口完全安静
• 像没烧录一样
• 换介质后彻底没反应

而这一层又偏偏是日志最少的一层。

**启动越早期，现象越沉默。**

## 最后怎么一句话记住？

如果你在面试、分享或者写文章时，想快速把这件事讲明白，可以直接用这句：

**BootROM 会根据芯片硬件启动配置，选择 SD、eMMC、SPI NOR、NAND 等启动介质之一，并按对应规则找到第一阶段镜像，把系统从"硬件上电状态"拉到"第一段软件开始执行"。**

如果再压缩成一句最容易记的话，就是：

**BootROM 定起点，介质决定规则。**

## 结尾

很多人一看启动问题，就会先去盯 U-Boot、看 bootcmd、看内核日志。

这当然没错，因为这些层最容易观察、也最容易实验。

但如果真想把"系统为什么能开始跑"这件事讲明白，

就一定要把视线再往前推一步。

BootROM 早就已经决定了两件最关键的事：

• **从哪种介质启动**
• **按什么规则找第一阶段镜像**

后面你能不能看到 U-Boot，前提都是：

**BootROM 先把第一棒交出去了。**
