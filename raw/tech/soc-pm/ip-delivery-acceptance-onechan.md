## 核心观点

IP 交付不是"把代码扔过去就完事"，而是一份**信任契约**。交付物质量直接决定后续集成工作的效率，甚至项目的成败。

## 交付物清单

### 1. 硬件交付物

| 交付物 | 内容 | 用途 |
|--------|------|------|
| RTL 代码 | 可综合的 Verilog/VHDL | 集成到 SoC |
| 综合脚本 | 约束文件、TCL 脚本 | 确保综合结果正确 |
| 仿真模型 | 行为模型、时序模型 | 系统级仿真 |
| 物理视图 | LEF、GDS | 物理实现 |
| 时序库 | Liberty 文件 | 时序分析 |
| 功耗库 | 功耗模型 | 功耗分析 |
| 测试向量 | ATPG 向量、BIST 向量 | 量产测试 |

### 2. 软件交付物

| 交付物 | 内容 | 用途 |
|--------|------|------|
| 驱动代码 | 底层驱动、HAL | 软件开发 |
| 固件代码 | BootROM、固件 | 系统启动 |
| 测试代码 | 自测试代码 | 验证 |
| 示例代码 | 使用示例 | 快速上手 |
| API 文档 | 接口说明 | 开发参考 |

### 3. 文档交付物

| 交付物 | 内容 | 用途 |
|--------|------|------|
| 设计文档 | 架构、接口、时序 | 理解设计 |
| 验证文档 | 验证计划、报告 | 确认质量 |
| 集成指南 | 集成步骤、注意事项 | 指导集成 |
| 用户手册 | 使用方法、配置 | 最终用户 |
| 勘误表 | 已知问题、workaround | 规避风险 |

## 验收标准

### 功能验收
- 所有功能点已验证
- 边界条件已测试
- 错误处理已验证
- 性能指标已达标

### 质量验收
- 代码覆盖率 ≥ 95%
- 功能覆盖率 100%
- 断言通过率 100%
- 无关键/高危 bug

### 文档验收
- 接口定义完整
- 时序参数准确
- 配置选项清晰
- 使用示例可运行

## 常见交付陷阱

### 陷阱 1：RTL 与模型不一致
行为模型和 RTL 实现存在差异，导致系统仿真通过但芯片失败。

### 陷阱 2：时序约束不完整
缺少关键路径约束，综合后时序违例在集成阶段才发现。

### 陷阱 3：功耗模型不准确
功耗估算偏差过大，导致电源设计不足或过热。

### 陷阱 4：文档与实现不同步
文档描述的是旧版本实现，实际代码已修改。

### 陷阱 5：缺少边界条件说明
未说明 IP 的工作边界（温度、电压、频率），集成时超规格使用。

## 验收检查清单

```
□ RTL 代码可综合，无 latch 推断
□ 仿真模型与 RTL 功能一致
□ 时序约束完整，关键路径已标识
□ 功耗模型已校准，与实测偏差 < 20%
□ 所有寄存器已文档化，读写权限明确
□ 中断/错误处理机制已验证
□ BootROM/固件已提供且可运行
□ 驱动代码已通过基本功能测试
□ 集成指南包含完整步骤和已知问题
□ 勘误表已更新至最新版本
```
