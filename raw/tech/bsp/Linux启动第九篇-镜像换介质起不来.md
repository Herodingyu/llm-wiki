很多人做板级 bring-up 时，都遇到过这种情况：

同样一份镜像，

烧到 SD 能启动，换到 eMMC 不行；

SPI NOR 上能起来，换到 NAND 就没反应。

这时候最容易冒出来的判断是：

• 镜像坏了
• 烧录错了
• U-Boot 配置有问题

这些都可能是原因。

但更常见的真相其实是：

**不是镜像内容变了，而是启动规则变了。**

## 一句话先讲明白

很多人理解的"同一个镜像"，通常只是：

• 文件名一样
• 内容一样
• 校验值一样

但系统启动看得远不止这些。

它真正关心的是：

• 镜像放在哪
• BootROM 能不能找到
• 前一阶段能不能继续往下读
• 后面的加载路径是不是也跟着改了

所以问题的本质不是：

**文件是不是同一个。**

而是：

**这条启动链在新介质上是不是还成立。**

## 第一，换介质，最先变的不是镜像，而是读取规则

很多人会觉得，

无非就是把同一个 bin 从一个地方挪到另一个地方。

但对启动链来说，换介质往往意味着：

• 读取方式变了
• 镜像位置变了
• 镜像格式要求可能变了
• 后续加载路径也可能变了

比如：

• SD / eMMC 更像块设备读取
• SPI NOR 更像固定地址读取
• NAND 还要考虑 ECC、坏块、page / block 规则

所以你看到的是"换存储介质"，

BootROM 和后续阶段看到的是：

**整条最早期加载规则都变了。**

**启动介质一换，第一跳的规则通常也跟着换。**

## 第二，为什么 SD 能起，换 eMMC 就不行？

这是最常见的一类问题。

因为很多人会把 eMMC 当成"板载 SD"，

但启动时两者经常不是一回事。

最典型的区别就在这里：

• SD 通常按卡的固定区域读
• eMMC 可能分 user area、boot0、boot1

也就是说，

你把镜像写进 eMMC 了，不等于 BootROM 就会去那里读。

很常见的情况是：

• 镜像写到了 user area
• 但 BootROM 默认从 boot0 读
• 结果就是完全起不来

所以这类问题最常见的根因，不是没烧进去，

而是：

**烧进去了，但没烧到启动链真正会去读的地方。**

## 第三，为什么 SPI NOR 能起，换 NAND 就容易翻车？

因为 NAND 比 SPI NOR 复杂得多。

SPI NOR 启动一般比较直接：

• 固定地址
• 线性读取
• 没有坏块
• 不依赖复杂 ECC

但 NAND 不一样，它往往涉及：

• page / block
• ECC
• 坏块
• OOB
• 特定镜像布局

所以很多时候，

不是镜像逻辑坏了，

而是：

**NAND 上的镜像布局不符合 BootROM 或早期加载程序的读取规则。**

**NAND 启动最难的，不是写进去，而是按它能认的方式写进去。**

## 第四，问题不一定卡在第一跳，也可能卡在后面几跳

还有一种很容易被忽略的情况是：

第一阶段其实已经起来了，

但后面的路径还没改过来。

比如：

• SPL 还在按旧介质找 U-Boot
• U-Boot 还在按旧路径找 kernel
• bootcmd 还是老配置
• bootargs 里的 root= 还指向原来的设备

所以"换介质起不来"不一定都卡在 BootROM。

也可能是：

• 第一跳对了
• 第二跳错了
• 或者 Linux 最后挂 rootfs 还在找旧设备

**前面一层介质改了，后面每一层的加载路径都要重新核对。**

## 第五，现场最常见的几个真实原因

把这类问题压缩一下，最常见的根因基本就这几种：

**镜像写入位置错了**

偏移不对、分区不对、区域不对

**镜像格式不对**

当前平台不是认裸 bin，而是认特定 header 或打包格式

**后续加载路径没改**

SPL、U-Boot、Kernel 还在按旧介质路径继续读

**设备编号变了**

比如 mmc 0 和 mmc 1 的角色变了

**介质特性没处理对**

eMMC 的 boot partition、NAND 的 ECC / 坏块规则

所以这类问题最怕的不是复杂，

而是：

**你只盯着镜像文件本身。**

## 第六，最实用的排查思路是什么？

这篇文章最值得收藏的，其实就是这句话：

**先判断系统走到了哪一层，再判断为什么没往下走。**

你可以按这个顺序查：

1. **先看最远走到哪**

• 完全没输出：先怀疑 BootROM、启动模式、第一阶段镜像位置
• 有 SPL，没 U-Boot：先看 SPL 读下一阶段是否正常
• 有 U-Boot，没内核：先看 bootcmd、kernel、dtb 加载路径
• 内核起来了，挂不住 rootfs：先看 root=、分区、驱动和设备节点

2. **再看当前介质是不是你以为的那个**

有时候不是"没启动"，

而是系统根本没按你以为的介质启动。

3. **核对每一跳的读取路径**

顺着这条链去看：

• BootROM 读谁
• SPL 读谁
• U-Boot 读谁
• Kernel 最后挂谁

这比反复重烧镜像更有效。

**启动排查最有效的方法，不是胡乱改配置，而是顺着"谁加载谁"一层层查。**

## 最后怎么一句话记住？

如果你在面试、分享或者文章里想快速讲清楚，可以直接用这句：

**同一个镜像换个启动介质起不来，很多时候不是镜像内容错了，而是新介质上的启动规则、镜像位置和后续加载路径没有同步匹配。**

如果再压缩成一句最容易记的话，就是：

**镜像可以一样，启动规则未必一样。**

## 结尾

很多人第一次遇到这类问题，会觉得很玄。

其实一点也不玄。

因为启动从来不是"文件对了就行"，

而是前一阶段要能正确找到下一阶段。

所以换启动介质之后，真正要检查的不是：

**这份镜像还是不是同一个。**

而是：

**这条启动链在新介质上是不是还成立。**
