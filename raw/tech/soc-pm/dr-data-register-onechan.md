# 数据寄存器 DR 的本质：不是存储单元，而是数据搬运通道

> 来源：微信公众号「OneChan」
> 原文链接：https://mp.weixin.qq.com/s/L7w6thW5h2dCwyfB80VPmw
> 记录时间：2026-05-04

## 核心观点

**DR 根本不是一个"存储"数据的地方，它是 CPU 总线和外设内部硬件之间唯一的"数据搬运通道"**。对 DR 执行的任何一次读或写，本质上都是在**触发一次硬件级别的数据传输事务**。

## DR 的物理本质：双缓冲结构的"透明窗口"

绝大多数现代 MCU 的外设 DR，在物理上对应着**两个完全独立的硬件寄存器**：
- **TDR**（Transmit Data Register，发送数据寄存器）
- **RDR**（Receive Data Register，接收数据寄存器）

但它们**共享同一个内存地址**，对外只暴露一个名字 DR。

- **写 DR** → 实际写入 **TDR**
- **读 DR** → 实际读取 **RDR**

你永远无法通过软件读取到刚刚写入 TDR 的值，这两个寄存器之间完全隔离，只有外设硬件逻辑才能在它们之间移动数据。

## DR 的核心工作机制：与移位寄存器的协同

### 发送过程
1. 软件写 DR → 数据写入 TDR，TXE 立即清零
2. 硬件搬运 → 移位寄存器为空时，自动将 TDR 数据并行加载到移位寄存器
3. TXE 置位 → 表示 TDR 已空，可写入下一个数据
4. 串行发送 → 移位寄存器按波特率一位一位从 TX 引脚输出
5. TC 置位 → 最后一位发送出去**且 TDR 也为空**时，TC=1

**关键细节**：TXE=1 只表示"TDR 空了"，**不代表"数据发送完成"**。看到 TXE=1 时，数据很可能还卡在移位寄存器里。

### 接收过程
1. 串行接收 → RX 引脚检测到起始位，按波特率采样移入移位寄存器
2. 硬件搬运 → 一帧完整接收后，自动并行加载到 RDR
3. RXNE 置位 → 表示有数据待读取
4. 软件读 DR → CPU 读取 RDR，RXNE 立即清零

**关键细节**：RXNE=1 时新帧接收完成，若 RDR 仍保留前一帧数据，会触发 **ORE（过载错误）**，新数据直接丢失。

## 不同外设 DR 的设计差异

| 外设 | DR 结构 | 读写行为 | 核心设计考量 |
|------|---------|----------|-------------|
| UART | 独立 TDR+RDR，单缓冲 | 写→TDR，读→RDR | 异步通信，收发速率可能不匹配 |
| SPI | 独立 TDR+RDR，双缓冲，收发捆绑 | 写→加载发送缓冲，读→取出接收缓冲 | 同步全双工，收发必须同频同相 |
| I2C | 独立 TDR+RDR，带地址匹配 | 写→发送数据/地址，读→接收数据 | 多主机、主从架构 |
| ADC | 单 RDR（数据保存器），多通道共享 | 只读，写无效 | 单向转换，结果具有时效性 |
| DAC | 单 TDR 或带保持寄存器的双缓冲 | 只写，读通常无效 | 单向输出，双缓冲消除更新抖动 |

## 最常见开发坑

### 坑1：误将 TXE 当作 TC 使用
```c
// 错误：最后一个字节还在移位寄存器中，并未真正发出！
while(!(USART1->SR & USART_SR_TXE));
USART1->DR = data;
USART1->CR1 &= ~USART_CR1_UE;  // 串口关闭，数据被截断

// 正确：等待 TC（发送完成）
while(!(USART1->SR & USART_SR_TC));
USART1->CR1 &= ~USART_CR1_UE;
```

### 坑2：读 DR 时没有先检查 RXNE
中断服务函数中不区分中断源就读 DR，可能读到垃圾数据并错误清除 RXNE，导致后续正确接收的数据被饿死。

### 坑3：FIFO 模式下只读取一次 DR
触发一次 RXNE 中断只读一次 DR 就退出，FIFO 中剩余数据仍使 RXNE 保持置 1，立即再次触发中断，形成中断风暴。

## DR 的设计哲学

芯片设计师把 DR 搞成"反直觉"的样子，目的是：**用最小的 CPU 干预代价，换取外设最大的硬件自主管线和并行度**。

DR 的双缓冲实现了 **CPU 执行域和串行外设域之间的并行流水**：
- 发送时：外设还在移位发送上一帧，CPU 已将下一帧写入 TDR
- 接收时：CPU 正在处理上一帧，硬件已完成下一帧接收放在 RDR 中等待

## 总结

DR 是：
- CPU 总线和外设硬件之间的**唯一数据接口**
- 软件世界与硬件时序世界之间的**刚性边界**
- 并行 CPU 域与串行外设域之间的**无缝转换器**
- 实现 CPU 自主性与外设自主性**并行驰骋**的关键机制

> 理解 DR 的本质，你就理解了所有外设数据通信的精髓。
